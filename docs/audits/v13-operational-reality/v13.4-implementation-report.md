## Problem addressed

V13.3 drill (2026-06-08) surfaced a launch blocker: **Supabase Free Plan has no automated backups.** The database could be lost with no recovery path. Without a backup mechanism, the launch readiness checklist had an un-closeable gap.

---

## What was built

### Backup system

| Component | File | What it does |
|---|---|---|
| Shared library | `scripts/backup-lib.mjs` | Pure functions: encrypt/decrypt, checksum, manifest, validation |
| Backup script | `scripts/backup-production.mjs` | Exports 8 core tables via REST API, encrypts, writes manifest + checksums |
| Verify script | `scripts/verify-latest-backup.mjs` | 10-check gate: archive present, checksum match, decrypts, core tables, fresh (<24h), PRODUCTION env |
| Restore script | `scripts/restore-backup-local.mjs` | Decrypts + restores to throwaway Supabase; seeds branches/product_categories/suppliers/pickup_windows from source; creates auth users via GoTrue admin API; upserts in FK-safe order |
| Parity script | `scripts/verify-restore-parity.mjs` | Service-role-only parity + integrity check; no test users needed |
| Cert script | `scripts/generate-v134-certification.mjs` | Generates signed `disaster-recovery-certification.md` |
| GitHub Actions | `.github/workflows/production-backup.yml` | Daily cron at 02:00 UTC + manual trigger; uploads encrypted artifact (90-day retention) |
| Runbook | `docs/runbooks/free-tier-backups.md` | Why/how/quarterly drill steps for the non-technical owner |
| Unit tests | `src/lib/server/backup-system.test.ts` | 31 tests covering encrypt/decrypt, manifest, validation, checksums |

### Encryption scheme

```
[8 bytes PLAICE\x01\x00 magic]
[32 bytes scrypt salt]
[16 bytes AES-GCM IV]
[16 bytes AES-GCM auth tag]
[variable: gzip(JSON payload) encrypted with AES-256-GCM]
```

Key derivation: scrypt N=16384, r=8, p=1. Wrong key or tampered ciphertext → decryption throws (auth tag mismatch).

### No pg_dump required

The backup uses the Supabase REST API (`/rest/v1/{table}`) with the service role key to export table data as JSON. This works on any OS with no database tools installed.

## Known limitations (free-tier constraints)

| Limitation | Impact | Mitigation |
|---|---|---|
| `auth.users` not backed up | After restore, staff must reset passwords | Auth sessions are ephemeral; orders/inventory/audit data is intact |
| `updated_at` housekeeping drift | Trivial — set_updated_at trigger re-fires on restore | Integrity check excludes `updated_at` (business data columns verified) |
| Backup max age ~24h | Up to 1 day of transactions lost in catastrophic failure | Acceptable for free-tier; upgrade to Supabase Pro for PITR if needed |
| GitHub artifact 90-day retention | Artifacts older than 90 days are deleted | Run quarterly drill + download and archive if longer retention needed |
| No pg_dump = no schema backup | Schema must be reconstructed from migrations | All schema is in `supabase/migrations/` — migrations applied automatically during restore |
