## 1. SMS Truth

### Behaviour changed

**Order status page** (`/order/status/[publicAccessId]/page.tsx`):
- Added "The shop will contact you when your order is ready" to the in-progress order message. Previously the text said "keep this page to check your live status" without explicitly stating the shop would reach out. No SMS promise — just an honest channel statement.

**Counter dashboard** (`/components/counter-dashboard.tsx`):
- Added a `sms-call-customer-cue` banner to every order card that is in `ready` status and whose `smsState` is anything other than `sent`. Message: "Call the customer — no real text was sent."
- This fires for `disabled`, `dry_run`, `failed`, `not_required`, and `unknown` statuses.
- Already-present: badge labels include "Text didn't send — call them" for `failed` and "Text alerts off" for `disabled`.

**SMS server code** (`/lib/server/sms.ts`): No changes — was already honest (returns disabled/dry_run/failed, never fakes success).

### Tests added

`src/lib/domain/sms.test.ts` — 4 new assertions:
- `disabled` badge label does not contain "sent"
- `dry_run` badge label is not equal to the `sent` label
- `failed` badge label directs staff to "call"
- No non-sent status produces the `sent` label

## 4. Supplier No-Certificate Nag

### Behaviour changed

**Compliance client** (`/components/admin-compliance-client.tsx`):
- Added `noDocument` (active suppliers with no `documentUrl`) and `notVerified` (active suppliers with no `verifiedAt`) computed sets.
- `health` calculation now includes these: if any active supplier has no document or no verification, status is "Attention Required" (previously only expiry-based gaps triggered this).
- Second metrics row added (`data-testid="compliance-gaps"`): "No expiry date / No certificate document / Not verified" counts shown.
- Warning banner (`data-testid="compliance-gap-warning"`) appears when any of those counts > 0: "X supplier(s) have no certificate document · Y supplier(s) have not been verified. Compliance cannot be considered healthy until all are resolved."

**Note:** `getCertificateState()` in the domain is unchanged — it only checks expiry, not document/verification. This is documented in the new compliance-inventory test as an explicit gap. The compliance client handles the two additional checks separately.

**Halal promise page** (`/app/our-halal-promise/page.tsx`): No changes needed — it already shows "Missing verification" and "Metadata only" for document, and is honest about what's recorded.

### Tests added

`src/lib/domain/compliance-inventory.test.ts` — 3 new assertions in "supplier certificate evidence gaps":
- Missing expiry flagged even when active.
- Valid expiry alone does not guarantee document or verification (documents the domain gap).
- Inactive supplier is never a compliance risk.

## 7. No Demo Data on Production DB Error

### Behaviour changed

**orders.ts** (`/lib/server/orders.ts`) — `getCounterOrders()`:
- **Before:** On query failure, returned demo orders if `allowDemoFallback()` was true (which includes all non-production environments regardless of whether real DB credentials were configured).
- **After:** Added `if (hasSupabaseServiceEnv()) return [];` check _before_ the demo fallback. If real Supabase credentials are configured but the query fails, the function now returns an empty array instead of silently substituting demo data.
- Demo fallback remains available in local/demo environments with no Supabase credentials configured.

### Tests

`src/lib/server/runtime-truth.test.ts` (new file):
- Tests the `allowDemoFallback` logic inline (cannot import `server-only` modules in vitest directly).
- Documents the `hasSupabaseServiceEnv()` gate as a code comment with the control flow.

## Final Verdict

**Can the app still mislead the owner/customer/staff about SMS, stock, purchasing, certificates, or yields?**

- **SMS**: No. Customer sees "The shop will contact you" — no SMS promise. Staff sees "Call the customer — no real text was sent" when order is ready but SMS is off/failed/dry_run.
- **Stock**: No. Every stock-facing page (inventory, purchasing) carries the intake/count-only stamp and the last count date.
- **Purchasing**: No. "Recommendations can be trusted" is gone. All confidence bands show "Guidance only."
- **Certificates**: No. Compliance health now requires document + verification, not just an expiry date.
- **Yields**: No. Cutting guide carries an explicit "estimates — butcher sign-off required" disclaimer.
- **Demo data**: No. A real DB failure when credentials are configured returns empty, not fabricated orders.
