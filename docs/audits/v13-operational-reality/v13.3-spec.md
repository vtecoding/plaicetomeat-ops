_Branch: `v13-reality-proof`. Spec date: 2026-06-08._

---

## Purpose

V13.3 closes the single highest remaining launch risk:

> **If the production database were corrupted or lost today, could we recover it?**

The app is technically ready (V12–V13.2). The missing evidence is not a code gap — it is a
human-and-process gap. Until a real operator has restored the real production backup into a real
non-production database and confirmed the data came back intact, we cannot sign off on launch.

V13.3 provides:
1. The exact steps to run the drill.
2. The tooling that captures evidence automatically.
3. The report format that constitutes a signed drill result.
4. The launch checklist gate that blocks go-live until the drill is passed.

## 1. Backup Source of Truth

The authoritative backup source is **Supabase managed backups** for the production project.

**Operator: before running the drill, confirm:**

1. Supabase dashboard → Project → Database → **Backups**.
2. A daily backup is listed dated **within the last 24 hours**. Record the timestamp.
3. If on a PITR-capable plan, PITR is enabled and the retention window is noted.
4. Production secrets are recorded in the team password manager
   (`SUPABASE_SERVICE_ROLE_KEY`, `ORDER_ACCESS_SECRET`, `STAFF_SESSION_SECRET`,
   `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `CANONICAL_BRANCH_ID`).

Reference: `docs/runbooks/backup.md`.

## 2. Restore Into a Non-Production Supabase Project

**Never restore over production.** All drills use a fresh, throwaway Supabase project.

**Operator steps:**

1. Create a **new** Supabase project (free tier is fine for a drill).
   Name it something obvious like `ptm-restore-drill-YYYYMMDD`.
2. Supabase dashboard (production project) → Database → Backups.
3. Select the latest daily backup (or a PITR point within the retention window).
4. Click **Restore** and select the throwaway project as the target.
   - Supabase will copy the backup data into the throwaway project's database.
5. Wait for the restore to finish and the database to report healthy in the dashboard
   (usually 5–15 minutes).
6. From the throwaway project, copy:
   - Project URL (`RESTORED_SUPABASE_URL`)
   - Service role key (`RESTORED_SUPABASE_SERVICE_ROLE_KEY`)

Reference: `docs/runbooks/restore.md`.

## 3. Verify Core Tables After Restore

Run the automated parity and integrity check against **both** the production source and the
restored target. The script compares row counts and spot-samples key records.

```
RECOVERY_ENVIRONMENT=PRODUCTION \
STRICT=1 \
SOURCE_SUPABASE_URL=<production url> \
SOURCE_SUPABASE_SERVICE_ROLE_KEY=<production service role key> \
RESTORED_SUPABASE_URL=<throwaway url> \
RESTORED_SUPABASE_SERVICE_ROLE_KEY=<throwaway service role key> \
SUPABASE_URL=<local or production url> \
SUPABASE_SERVICE_ROLE_KEY=<local or production key> \
node scripts/verify-disaster-recovery.mjs
```

**Tables verified (parity):**

| Table | What it holds |
|---|---|
| `profiles` | Staff accounts |
| `orders` | Customer orders |
| `order_items` | Line items per order |
| `products` | Product catalogue |
| `inventory` (batches) | Stock on hand |
| `audit_logs` | Compliance and security audit trail |
| `compliance_logs` | Temperature and compliance readings |
| `pricing_validations` | Butcher sign-off records |

**Integrity samples verified:** latest order, oldest order, random order, latest audit event,
oldest audit event, latest compliance log, latest pricing validation.

**Expected output:**

```
RESULT: disaster-recovery verification PASSED (RECOVERY_CERTIFIED)
```

If the output is `RECOVERY_FAILED` or `PARITY_FAILED`, the restore did not complete correctly or
the backup is inconsistent. Do not proceed to sign-off until the cause is identified and fixed.

## 4. Verify RLS Still Blocks Cross-Branch Access

RLS enforcement survives across restore — the policies are schema-level and travel with the
backup. The `verify-disaster-recovery.mjs` script already proves one key RLS control:

> `direct-write bypass blocked` — a manager-session client cannot directly insert into
> `recovery_artifacts` (RLS rejects it; only the SECURITY DEFINER RPC can write this table).

For broader RLS confidence, also run against the restored project's credentials:

```
SUPABASE_URL=<throwaway url> \
SUPABASE_SERVICE_ROLE_KEY=<throwaway service role key> \
node scripts/verify-audit-authenticity.mjs
```

```
SUPABASE_URL=<throwaway url> \
SUPABASE_SERVICE_ROLE_KEY=<throwaway service role key> \
node scripts/verify-checkout-integrity.mjs
```

Both must pass. `verify-audit-authenticity` proves `audit_logs` is not directly writable by
`anon`/`authenticated`. `verify-checkout-integrity` proves the checkout RPC rejects forged prices.

## 5. Verify Admin Can Still Load Orders, Inventory, Compliance

Point a running app instance (local or staging — **not production**) at the throwaway project's
credentials, then perform the manual smoke test from `docs/runbooks/restore.md §Post-restore
validation`:

1. Staff login succeeds (`/login`).
2. `/admin/today` loads without error.
3. `/admin/inventory` shows existing batches (row counts match the parity results).
4. `/admin/compliance` shows supplier records.
5. `/shop` lists products (storefront loads).
6. A TEST checkout completes end to end (`CHECKOUT_TEST_MODE_ENABLED=true`). The order appears
   at `/counter`. An audit row is written.

All six must pass. If any step fails, the restore is incomplete or the schema needs migration
forward (see `restore.md §Post-restore validation` step 1 for migration parity check).

Also run:

```
PRODUCTION_READINESS_MODE=strict \
SUPABASE_URL=<throwaway url> \
SUPABASE_SERVICE_ROLE_KEY=<throwaway service role key> \
node scripts/verify-production-readiness.mjs
```

Migration parity and secrets must pass.

## 6. Produce Signed Recovery Drill Report

After `verify-disaster-recovery.mjs` completes, note the drill ID printed to stdout (or query
`recovery_drills` for the latest row). Then generate the certification report:

```
RECOVERY_DRILL_ID=<drill id from step 3> \
STRICT=1 \
SUPABASE_URL=<same credentials as step 3> \
SUPABASE_SERVICE_ROLE_KEY=<same credentials as step 3> \
node scripts/disaster-recovery-certification.mjs
```

The report is written to `docs/reports/disaster-recovery-certification.md`.

**A valid V13.3 certification report MUST begin with:**

```
REAL PRODUCTION RECOVERY DRILL
```

A report beginning with `LOCAL TEST DATA ONLY` is **not valid** for launch certification.

The report records:
- Environment, drill type, operator email, timestamps
- Backup artifact name, size, and checksum
- Parity results table (source vs. restored row counts per table)
- Integrity sample results
- Final verdict: `RECOVERY CERTIFIED` or `RECOVERY FAILED`

Commit the updated `docs/reports/disaster-recovery-certification.md` to the main branch as part
of the V13.3 seal commit.

## Operator Note

The bottleneck is yours: a Supabase backup restore into a throwaway project takes approximately
10–20 minutes end to end. The scripts do the rest. The cost of skipping this step is going live
with no tested recovery path — which means any data loss event becomes an untested emergency.

Run the drill once before launch. It confirms the backup is not corrupt. It confirms you know
how to use the dashboard to restore. It confirms the app works after recovery. Those three things
are worth 20 minutes.
