## 2. Session Map

End-to-end trace: **login → middleware → cookies → auth helpers → route guards → role checks.**

### 2.1 Login
- Entry: [`loginAction`](src/app/actions/auth.ts:36) (server action, called from [`login-form.tsx`](src/components/login-form.tsx)).
- Flow:
  1. Normalises email, reads password + `returnTo` ([auth.ts:37](src/app/actions/auth.ts:37)).
  2. Generic error string for all failure modes — no user enumeration ([auth.ts:44](src/app/actions/auth.ts:44)).
  3. Captures client IP via `x-forwarded-for` / `x-real-ip` ([auth.ts:22-34](src/app/actions/auth.ts:22)) — **captured but unused for lockout** (see §5, F7).
  4. `isLoginLocked(email)` consults `login_attempts` ([auth.ts:56](src/app/actions/auth.ts:56) → [login-attempts.ts:14](src/lib/server/login-attempts.ts:14)). **Account-only** lockout.
  5. `supabase.auth.signInWithPassword` against an anon SSR client ([auth.ts:67](src/app/actions/auth.ts:67)).
  6. Profile re-read via **service-role** client to confirm `role` + `is_active` ([auth.ts:80-97](src/app/actions/auth.ts:80)); inactive/role-less ⇒ `signOut()` + generic error.
  7. `recordLoginAttempt` (success/failure) ([auth.ts:100](src/app/actions/auth.ts:100)).
  8. Seeds the idle-timeout cookie `ptm_staff_last_seen` ([auth.ts:106-112](src/app/actions/auth.ts:106)).
  9. `redirect(resolvePostLoginPath(role, returnTo))` ([auth.ts:115](src/app/actions/auth.ts:115)).
- **PII in logs**: failure paths log `email` and the entire `profile` object via `console.error` ([auth.ts:70-76](src/app/actions/auth.ts:70), [auth.ts:88-92](src/app/actions/auth.ts:88)). Violates "Remove PII from logs."

### 2.3 Middleware
- [`middleware.ts`](src/middleware.ts), matcher `["/counter/:path*","/admin/:path*","/compliance/:path*"]` ([middleware.ts:93](src/middleware.ts:93)).
- Steps: short-circuit non-staff paths ([:11](src/middleware.ts:11)) → require public env or redirect home ([:18](src/middleware.ts:18)) → idle-timeout check via `isStaffSessionExpired(lastSeen)` ([:43](src/middleware.ts:43)) → `getUser()` ([:52](src/middleware.ts:52)) → profile `role,is_active` read ([:60](src/middleware.ts:60)) → `canAccessStaffPath` ([:66](src/middleware.ts:66)) → **slide** `last_seen` forward every request ([:70-75](src/middleware.ts:70)).
- Unauthorised outcomes are **silent redirects** to `/` ([:80-82](src/middleware.ts:80)) or `/login` ([:84-90](src/middleware.ts:84)). No `/unauthorised`.

### 2.4 Auth helpers
- [`getCurrentProfile`](src/lib/server/auth.ts:34) — request-cached; `getUser()` then profile read on the **anon** client; returns `null` for unauthenticated/inactive/role-less; `branchId` is nullable and returned as-is. This is the canonical server-side identity read.
- [`createSupabaseServerClient`](src/lib/supabase/server.ts:13) anon SSR; [`createSupabaseServiceClient`](src/lib/supabase/server.ts:63) RLS-bypassing (used by login profile read, login-attempts, `getPublicBranch`).

### 2.5 Route guards & role checks
- **Domain logic**: [`canAccessStaffPath`](src/lib/domain/route-access.ts:30), [`isOwnerOnlyPath`](src/lib/domain/route-access.ts:26) (`/admin/releases`, `/admin/audit`), [`MANAGER_ROLES`](src/lib/domain/route-access.ts:6) = `["manager","owner"]`.
- **Page guards**: each admin page re-checks `getCurrentProfile()` + `MANAGER_ROLES.includes(role)` then `redirect("/")`. Owner-only pages [`/admin/audit`](src/app/admin/audit/page.tsx:19) and [`/admin/releases`](src/app/admin/releases/page.tsx:13) **use `MANAGER_ROLES`, not an owner check** — they rely on middleware to keep managers out (F2).
- **Action guards**: duplicated `requireManager`/`requireOwner`/`requireBranchStaff` in [admin-settings](src/app/actions/admin-settings.ts:28), [admin-schedule](src/app/actions/admin-schedule.ts:35), [admin-products](src/app/actions/admin-products.ts:33), [compliance-inventory](src/app/actions/compliance-inventory.ts:47), [ops-capture](src/app/actions/ops-capture.ts:39), [releases](src/app/actions/releases.ts:12) (owner — correct), [counter](src/app/actions/counter.ts:40), and inline in [carcass-intake](src/app/actions/carcass-intake.ts:75). **No single authority path** (F1).

### 2.6 Session timeout semantics
- `STAFF_SESSION_TIMEOUT_MS = 4h` ([route-access.ts:1](src/lib/domain/route-access.ts:1)), enforced as a **sliding idle** window (refreshed every request at [middleware.ts:70](src/middleware.ts:70)). **No absolute timeout** — an actively-used session never expires (F5).
- [`isStaffSessionExpired`](src/lib/domain/route-access.ts:56) returns **`false` when `lastSeen` is missing** — deleting the cookie defeats idle timeout (F4). Supabase `getUser()` still gates real auth, so this weakens but does not remove identity enforcement.

## 3. Branch Authority Map

### 3.1 Branch resolution sources
1. `profile.branchId` from [`getCurrentProfile`](src/lib/server/auth.ts:34) — the **only legitimate** per-user branch.
2. [`getPublicBranch()`](src/lib/server/catalog.ts:96) — **"first active branch"** (`is_active`, `order created_at asc, limit 1`), or `demoBranch` if unconfigured. Intended for the single-branch public storefront, **leaking into admin authority** as a fallback.
3. [`demoBranch`](src/lib/data/demo.ts) — static demo id.

### 3.3 Owner-only pages
- [`/admin/audit`](src/app/admin/audit/page.tsx) and [`/admin/releases`](src/app/admin/releases/page.tsx): middleware restricts to `owner` via `isOwnerOnlyPath`, **but the page bodies check `MANAGER_ROLES`** (manager+owner). Middleware is the only thing keeping managers out (F2).
- Action layer for releases ([releases.ts:12](src/app/actions/releases.ts:12)) **correctly** requires `owner`. The audit page is read-only (`getRecentAuditEvents`) with no owner re-check.

### 3.4 Manager-only / staff-only pages
- Manager surfaces: all other `/admin/*` — guarded by `MANAGER_ROLES` at page + middleware.
- Staff surfaces: `/counter/*`. [`/counter`](src/app/counter/page.tsx) has **no page-level role guard** — relies entirely on middleware; branch comes from `profile?.branchId ?? demoBranch.id`.
- `requireBranchStaff(branchId)` ([counter.ts:40](src/app/actions/counter.ts:40)) enforces cross-branch denial for counter actions (owner exempt) — the one place branch-scoping is done correctly.

### 3.5 Dead authority config
- Middleware matches `/compliance/:path*` and `STAFF_ROUTES` includes `/compliance`, but **no `/compliance` app route exists** (only `/counter/compliance`, `/admin/compliance`). Dangling, low-risk (F8).

## 4. Trust Boundary Report

### 4.1 Current trust assumptions
- Supabase-issued auth cookie is the source of truth; always re-validated by `getUser()` (server-side) in both middleware and `getCurrentProfile`. **Sound.**
- `profiles.role` / `is_active` read fresh per request. **Sound.**
- Middleware is assumed to run for every `/admin`, `/counter`, `/compliance` request, and several authority decisions (counter role, owner-only page access) **depend on that assumption** rather than re-checking at the page (F2, counter).
- `ptm_staff_last_seen` is trusted as an integrity-bearing idle marker although it is unsigned (F4).

### 4.2 Fail-closed paths (good)
- Modified/expired Supabase JWT ⇒ `getUser()` returns no user ⇒ redirect to `/login`.
- Missing cookie ⇒ no user ⇒ login.
- Cross-user cookie ⇒ JWT bound to a different `sub`; profile read keyed by `user.id`; cannot impersonate.
- Inactive/role-less profile ⇒ `null` ⇒ refused (middleware, `getCurrentProfile`, all action guards).
- Counter cross-branch write ⇒ refused ([counter.ts:47](src/app/actions/counter.ts:47)).
- Login lockout after 5 failures/15 min ([domain/auth.ts:3](src/lib/domain/auth.ts:3)).

### 4.3 Fail-open / silent paths (to fix)
- **F4** Missing `ptm_staff_last_seen` ⇒ idle timeout skipped ([route-access.ts:56-58](src/lib/domain/route-access.ts:56)).
- **F3** Branch fallback to first-active/demo branch when `profile.branchId` is null (§3.2).
- **F2** Owner-only pages render for managers if middleware is bypassed (page checks `MANAGER_ROLES`).
- **F6** `logoutAction` swallows `signOut()` failure with `catch {}` ([auth.ts:123](src/app/actions/auth.ts:123)) — failure not surfaced; still clears local cookie + redirects.
- **F5** No absolute session lifetime (sliding-only).
- **F7** IP/network captured but lockout is account-only.
- **F9** PII (`email`, full `profile`) written to logs on login failure.

## 6. Proposed Implementation Plan (mapped to spec — pending approval)

Reusable building blocks already in the codebase: HMAC signing
([order-access-token.ts](src/lib/server/order-access-token.ts)), rate-limit
infra `check_rate_limit` + `clientNetworkHash` + `hashIdentity`
([rate-limit.ts](src/lib/server/rate-limit.ts)).

1. **Session Envelope** (F4, F5, F10): replace plaintext `ptm_staff_last_seen` with an HMAC-signed, timestamped, user-bound envelope `{ userId, issuedAt, lastSeen }`. Enforce both **idle** and **absolute** timeouts. Any signature/binding/expiry mismatch ⇒ `SESSION_INVALID` (sign out + `/unauthorised`), never silent acceptance. Missing envelope on a staff path ⇒ fail closed.
2. **Branch Authority Helper** (F1, F3): add `requireStaffContext({ role?, needsBranch? })` returning `{ profile, branchId }` or a typed refusal; **no fallback**. A null branch on a branch-scoped surface ⇒ `/unauthorised`. Migrate all admin pages + actions to it; delete duplicated guards and every `?? branch.id` / `?? demoBranch.id` / `?? ""`.
3. **Owner Revalidation** (F2): `/admin/audit` and `/admin/releases` page bodies check `role === "owner"` server-side (not `MANAGER_ROLES`); audit data read gated likewise.
4. **`/unauthorised` page**: explicit page; middleware + guards redirect there instead of silently to `/`.
5. **Logout Hardening** (F6): remove `catch {}`; surface `signOut()` failure to the user; still clear local state.
6. **Login Hardening** (F7, F9): extend lockout to account **+ IP + network hash** (reuse `clientNetworkHash`/`hashIdentity`); remove `email`/`profile` from logs (log hashed identity only).
7. **Remove Branch Fallbacks** (F3): eliminate every production-branch `??` (table §3.2); keep `getPublicBranch` for the **public storefront only**.
8. **Cleanup** (F8): drop the dead `/compliance` authority entries (or add the route) — decide during implementation.

## 8. Remaining Risks

- **Session theft of a live valid cookie** remains possible (inherent to cookie auth); mitigated by idle+absolute timeout and user-binding, not eliminated. Full device binding is out of V12.2 scope.
- **Migrating ~17 surfaces off the branch fallback** risks breaking single-branch operation if the real branch has `branch_id = null` profiles in prod. Requires confirming production profiles all carry a `branch_id` before deploy (operator step).
- **Absolute timeout** will force re-login for long sessions — a deliberate UX trade-off for a non-technical owner; needs sign-off on duration.
- **Rate-limit on IP/network** can cause false lockouts behind shared NAT; tune thresholds conservatively.
- RLS/SQL authority was **not executed** in discovery; the app-layer plan assumes existing RLS remains the backstop.
