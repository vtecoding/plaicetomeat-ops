# V12.8 Discovery - Observability

Date: 2026-06-07
Scope: V12.8 discovery only. No implementation, no migrations, no test edits.
Builds on: V12.0-V12.7.

## 0. Verdict

The system is functionally hardened (auth, checkout, inventory, audit, operational
truth) but it is **observationally opaque**. There is no unified structured log
format, no operational counters, no health endpoint, no synthetic readiness check,
and no alert path. Failures are handled correctly for the *caller* (fail-closed,
safe messages), but they are not consistently made *visible* to an operator.

V12.8 must add a thin observability layer over the existing hardened code:
structured logging, in-process metrics, explicit health states, an `/api/health`
endpoint, a synthetic health verifier, and a vendor-neutral alert abstraction.
No behavioural change to the hardened paths.

This is a foundation, not a telemetry platform. In-process counters are
per-runtime-instance and reset on cold start; durable time-series forwarding is a
later, provider-specific phase (deferred).

## 1. Logging ŌĆö current state

`console.*` is used ad hoc in 15 server files (21 call sites). It is already
PII-careful (auth logs `code`/`status` only, checkout logs `branchId` + message,
security-audit logs `reason` + message), but:

- there is **no shared format** ŌĆö each line is a bespoke string + object;
- there is **no severity/category taxonomy** ŌĆö everything is `console.error`;
- there is **no redaction guarantee** ŌĆö correctness depends on each author
  remembering not to log secrets;
- logs are **not correlatable** ŌĆö no consistent `branch_id` / request id field.

Representative sites:
- `src/app/actions/auth.ts` ŌĆö sign-in failure, inactive account, sign-out failure.
- `src/lib/server/orders.ts` ŌĆö `create_checkout_order` faults, unexpected RPC shape.
- `src/lib/server/security-audit.ts` ŌĆö audit emit failure (the observer's observer).
- `src/lib/server/operations-intelligence.ts`, `shop-intelligence.ts`,
  `purchasing-intelligence.ts`, `setup-checklist.ts`, `rate-limit.ts`,
  `order-access-*.ts`, `public-order-access.ts`, `launch-readiness.ts`,
  `staff-session.ts`, `audit.ts` ŌĆö assorted best-effort error logs.

## 2. Error handling ŌĆö current state

Error handling is deliberately fail-closed and caller-safe:
- checkout: business-rule fragments surfaced verbatim; unknown faults replaced
  with a generic message and logged (`src/lib/server/orders.ts`).
- ops-capture: `SAFE_PATTERNS` allowlist surfaces known plain-English failures;
  everything else falls back to a generic message (`src/app/actions/ops-capture.ts`).
- security-audit: `recordSecurityEvent` never throws ŌĆö a failed audit must not
  break the request it observes.

The gap is not correctness, it is **visibility**: an unknown checkout fault, a
stale stock-count rejection, a checklist completion failure, or an audit-emit
failure are logged inconsistently (or only as a generic caller message) and are
not counted, so an operator cannot see failure *rates*.

## 3. Audit coverage ŌĆö current state (V12.4, do not revisit)

`security_event` audit is live via `recordSecurityEvent` ŌåÆ hardened
`emit_audit_log` (service-role-only, allowlisted, size-capped, secret-stripped).
It covers failed login, lockout, session missing/expired/tampered/cross-user,
role/branch/no-branch/owner denial, unauthorised route, logout failure.

Observability should **complement** this, not duplicate it: audit = durable,
queryable, DB-side security evidence; metrics/logs = operational visibility of
rates and faults. The natural integration point is `recordSecurityEvent` itself
(a single chokepoint already crossed by auth + middleware).

## 4. Health checks ŌĆö current state

None. There is no `/api/health`. `get_migration_health()` exists in the DB
(V4 migration, granted to `anon`) and returns `(expected_version, migration_name,
applied)` rows for required migrations ŌĆö this is the canonical DB-side migration
parity signal and can back a health endpoint without filesystem access.

`scripts/check-migrations.mjs` performs release-time drift checking but is not a
runtime health signal.

## 5. Release reporting ŌĆö current state

`scripts/release-report.mjs` and `scripts/audit-bundle.mjs` exist for governance
bundles. These are build/governance artefacts, not runtime observability.

## 6. Background jobs ŌĆö current state

No standalone background workers. SMS dispatch is request-driven. There is no
queue/cron runtime to instrument in this app.

## 7. Subsystem instrumentation points (clean chokepoints)

| Counter | Site |
|---|---|
| `checkout_success` / `checkout_failure` | `createCheckoutOrder` in `src/lib/server/orders.ts` |
| `checkout_partial_success` | reserved ŌĆö order created but a secondary step (e.g. SMS) fails; no such path at create today |
| `login_success` | `loginAction` success in `src/app/actions/auth.ts` |
| `login_failure` | `recordSecurityEvent` reason map (`login_failed`, `login_locked_out`) |
| `authority_denied` | `recordSecurityEvent` reason map (`authority_denied_*`, `unauthorised_route`) |
| `inventory_stale_rejection` | `applyStockCountLine` `STALE_STOCK_COUNT` error in `src/app/actions/ops-capture.ts` |
| `checklist_completion_failure` | `completeChecklist` error in `src/app/actions/ops-capture.ts` |
| `rpc_denied` | unexpected permission/denied RPC errors (checkout) |
| `database_error` | unexpected (non-business) RPC faults + audit emit failure |

## 8. Silent-failure / swallowed-exception inventory

- audit emit failure: caught, logged, swallowed (correct ŌĆö never break the
  observed request). **Add:** `database_error` counter + structured log.
- checkout unknown fault: logged + generic caller message. **Add:** classify
  `rpc_denied` vs `database_error`, structured log, `checkout_failure`.
- ops-capture generic-message fallback: business errors are visible to the user
  but unknown faults are flattened. **Add:** counters at stale + completion sites.
- intelligence loaders (`operations-intelligence`, `shop-intelligence`,
  `purchasing-intelligence`): errors logged, degraded data returned (V12.7 truth
  states already cover the UI). Out of scope to re-plumb; structured logging may
  be adopted incrementally.

## 9. Missing telemetry / health / alerts

- missing: unified structured log layer with category/severity/redaction.
- missing: operational counters.
- missing: explicit health states + `/api/health`.
- missing: synthetic readiness check (`scripts/verify-health.mjs`).
- missing: alert abstraction (console / webhook / future Sentry).

## 10. Recommended V12.8 implementation shape

1. `src/lib/server/observability/log.ts` ŌĆö edge-safe structured logger.
   Categories: AUTH, CHECKOUT, AUDIT, INVENTORY, OPS_CAPTURE, RELEASE, SYSTEM.
   Every entry: timestamp, category, severity, message; `branch_id` / request id
   where relevant. Defensive redaction of secret-like keys and JWT-like values.
2. `src/lib/server/observability/metrics.ts` ŌĆö edge-safe in-process counters for
   the list in ┬¦7.
3. `src/lib/server/observability/alerts.ts` ŌĆö vendor-neutral dispatch: console
   sink always, webhook when `ALERT_WEBHOOK_URL` set, Sentry stub for the future.
4. `src/lib/domain/health.ts` ŌĆö `HealthState` = HEALTHY | DEGRADED | UNAVAILABLE
   | CONFIGURATION_REQUIRED, with a `worstState` combinator.
5. `src/app/api/health/route.ts` ŌĆö app + configuration + database + migration
   parity, plus the metrics snapshot. No secrets; generic DB detail in the body,
   full detail to the log.
6. `scripts/verify-health.mjs` ŌĆö synthetic: DB connectivity, migration parity,
   checkout readiness, and (when an app URL is reachable) home/shop/health-endpoint.
7. Minimal instrumentation at the clean chokepoints in ┬¦7 only.

## 11. Risk map

High: none ŌĆö additive layer over hardened code.
Medium: importing the new modules into Edge middleware (via `recordSecurityEvent`)
requires them to be edge-safe (no `server-only`, no node builtins) ŌĆö mirror the
existing `security-audit.ts` constraint.
Low: in-process counters are per-instance and reset on cold start ŌĆö documented
as a known limitation; durable forwarding deferred.

## 12. Stop / Go

GO for a narrow V12.8 implementation:
- structured logging, metrics, health states + `/api/health`, synthetic check,
  alert abstraction, and minimal chokepoint instrumentation.

STOP: discovery complete. Implementation may begin now.
