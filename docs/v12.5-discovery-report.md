# V12.5 Discovery — Inventory Concurrency Integrity

Date: 2026-06-07
Scope: V12.5 (Inventory Concurrency Integrity) **discovery only**. No implementation, no migrations, no test edits. No checkout changes.
Builds on: V12.1–V12.4 (RPC seal, identity/branch, checkout integrity, audit coverage).

---

## 0. Verdict (read this first)

**Inventory is in much better shape than feared. Writes are atomic, locked,
authorised, non-negative, and audited in-transaction; an immutable movements
ledger already exists. The real gap is a single stale-write at stock-count apply.**

- ✅ Every batch mutation takes `SELECT … FOR UPDATE` on the batch (pessimistic lock).
- ✅ Each RPC is one transaction: batch update + movement + (waste/count) evidence + audit are atomic.
- ✅ Negative stock is impossible (`CHECK remaining_weight_kg >= 0` + per-RPC validation).
- ✅ **Order fulfilment does NOT touch inventory** — no `SALE` movement is ever written (pay-on-collection, cut-to-order), so checkout never interleaves with stock.
- ✅ Reads fail to **empty**, not demo/fallback stock.
- ⚠️ **No optimistic concurrency / `row_version`.** Stock-count *apply* writes the
  record-time `counted_weight_kg` using only the record-time `system_weight_kg`
  snapshot — it never re-checks the batch's *current* value at apply time. A waste
  or manual adjustment made between counting and applying is **silently clobbered
  (lost update)**.
- ⚠️ Manual `admin_adjust_inventory_remaining` has no expected-value guard (smaller, owner-only).
- ⚠️ `ADJUSTMENT` movements store `abs(delta)` — the **sign is lost**, so the ledger alone isn't replayable.
- ⚠️ Product availability flags are **decoupled** from real batch stock (truth conflation).

**So V12.5 is a narrow CAS slice, not an inventory rebuild.** The fix is a
compare-and-set guard at apply (and optionally manual adjust), using the existing
batch lock — `system_weight_kg` is a ready-made expected-version baseline.

## 1. Inventory data model map

| Table / view | Role | Key columns | Branch |
|---|---|---|---|
| `inventory_batches` | **canonical stock** (one row per received batch) | `received_weight_kg`, `remaining_weight_kg` (`CHECK >= 0`), `status` (active/depleted/disposed/recalled), `product_id`, `expiry_date`, `cost_per_kg`, `manual_adjustment_reason` | `branch_id` (FK, NOT NULL) |
| `inventory_movements` | **append-only ledger** | `movement_type` (RECEIVED/SALE/WASTE/TRANSFER/ADJUSTMENT), `quantity_kg` (`CHECK > 0`), `batch_id`, `reference_id`, `reason`, `created_by` | `branch_id` |
| `inventory_waste_events` | waste detail | `waste_kg` (`>0`), `reason`, `batch_id`, `product_id` | via batch |
| `stock_count_lines` | count evidence | `system_weight_kg` (snapshot at record), `counted_weight_kg`, `correction_movement_id`, `applied_at`, **UNIQUE(session_id, batch_id)** | `branch_id` |
| `ops_checklist_sessions` | stock-count session container | `kind='stock_count'`, `status` (in_progress/…), `business_date` | `branch_id` |
| `stock_levels` (view) | per product/branch totals | `sum(remaining_weight_kg) WHERE status='active'` | grouped |
| `products` | catalogue + **manual** availability | `is_available`, `stock_status`, `min/max_order_quantity` | `branch_id` |

- **Product/catalog coupling:** batches/movements reference `products(id)`; the
  `stock_levels` view aggregates per product. **But `products.is_available` /
  `stock_status` are NOT derived from batches** — they're set manually (see §4 I4).
- **Branch coupling:** every inventory row carries `branch_id`; all RPCs check
  `is_branch_manager(branch_id)`; reads are RLS-scoped to branch staff.

## 2. Inventory mutation map

| Path | RPC (all SECURITY DEFINER, authenticated→manager-gated) | Effect | Lock | Ledger | Audit (in-txn) |
|---|---|---|---|---|---|
| Receive batch | `admin_create_inventory_batch` | INSERT new batch | n/a (insert) | RECEIVED | `stock_added`/`batch_received` |
| Carcass intake | `admin_confirm_carcass_intake` | creates per-cut batches | n/a | RECEIVED | `carcass_intake_confirmed` |
| Waste | `admin_record_inventory_waste` | `remaining -= qty` (≤ remaining) | `FOR UPDATE` batch | WASTE | `waste_recorded` |
| Manual adjust | `admin_adjust_inventory_remaining(batch, new, reason)` | `remaining = new` (0…received) | `FOR UPDATE` batch | ADJUSTMENT (`abs(delta)`) | `stock_corrected` |
| Count record | `ops_record_stock_count_line(session, batch, counted)` | snapshot `system_weight=remaining`, upsert line | **no lock** (snapshot) | — | `stock_count_recorded` |
| Count apply | `ops_apply_stock_count_line(session, line, reason)` | if counted≠system → calls `admin_adjust_inventory_remaining(batch, counted, …)` | `FOR UPDATE` line; batch locked inside adjust | ADJUSTMENT | `stock_count_line_applied` + `stock_corrected` |

- **Server actions** (`src/app/actions/compliance-inventory.ts`, `ops-capture.ts`, `carcass-intake.ts`) all go through `resolveStaffContext`/`requireManager` then call the RPCs. No app-layer direct table writes.
- **Direct table writes:** none from app code — all inventory writes are RLS-revoked to clients and flow through the SECURITY DEFINER RPCs.
- **Waste/write-off paths:** only `admin_record_inventory_waste` (+ `'review'` reason used by intake review). 
- **Stock-count paths:** record → apply (above).
- **Carcass/intake/yield:** create new batches only; never mutate an existing batch's `remaining` (so they don't race count/waste on existing rows).

## 3. Concurrency map

- **Transaction boundaries:** each RPC = one implicit transaction (plpgsql function). All effects atomic.
- **Locking:** pessimistic `SELECT … FOR UPDATE` on the batch for waste & adjust; on the line + (read) session for apply. Default READ COMMITTED.
- **Row-version / CAS:** **none.** No `row_version`/`xmin`/expected-value parameter on any inventory mutation.
- **Stale-write risk (the headline):** `ops_apply_stock_count_line` decides the correction from `counted_weight_kg` vs the **record-time** `system_weight_kg`, then sets the batch to `counted_weight_kg`. The batch lock is held only *during* the write, not since the count — so any waste/adjust between record and apply is overwritten. **Lost update.**
- **Interleaving matrix (existing-batch `remaining`):**
  - waste ↔ count-apply: **races (I1)** — apply clobbers the waste.
  - manual-adjust ↔ count-apply: **races (I1)**.
  - manual-adjust ↔ manual-adjust: **races (I2)** — second overwrites first (no expected value).
  - count-apply ↔ count-apply (same line): safe (FOR UPDATE + `applied_at` idempotency).
  - count-apply ↔ count-apply (different sessions, same batch): **races (I1)**.
  - intake/carcass ↔ anything: safe (creates new batches; doesn't touch existing `remaining`).
  - order fulfilment ↔ inventory: **decoupled** (no SALE decrement).

## 4. Truth-state map

| Concept | Where represented | Conflation risk |
|---|---|---|
| Real measured stock | `inventory_batches.remaining_weight_kg` (sum via `stock_levels`) | authoritative |
| Sellable on storefront | `products.is_available` + `stock_status` | **decoupled from batches (I4)** |
| Zero stock | batch `remaining=0` → status depleted/disposed | distinct from… |
| Unavailable | `products.is_available=false` (manual) | …conflated with zero stock at the UI: a product can be "available" with 0 batch kg, or "out of stock" with stock present |
| Uncounted | no `stock_count_line` for a batch this session | not surfaced as a distinct state |
| Missing data | reads return **empty** (no service env) | ✅ no demo/fallback stock injected (`getInventoryBatches`/`getSuppliers` return `[]`) |

- **UI surfaces:** `/admin/inventory` (batches + value-at-risk), `/admin/stock-count` (count screen), `/admin` Business Insights (low-stock/expiry signals derived from batches). Storefront/checkout availability reads **products flags**, not batch truth.
- **No demo/fallback stock** appears in production reads (a positive vs the V12.2 branch-fallback issue).

## 5. Audit / evidence map

- **Audited (all in-transaction, authoritative — see V12.4):** waste (`waste_recorded`), manual adjust (`stock_corrected`), count record (`stock_count_recorded`), count apply (`stock_count_line_applied` + nested `stock_corrected`), batch receive (`stock_added`/`batch_received`), carcass intake (`carcass_intake_confirmed`).
- **Not separately audited:** batch `status` auto-transition to depleted/disposed (captured implicitly in the movement/audit metadata `remaining_kg`).
- **In-transaction:** yes — every audit insert is inside the mutating RPC's transaction.
- **Replayability:** `inventory_movements` is append-only and covers every change, **but `ADJUSTMENT` stores `abs(delta)` — the sign is lost (I3)**, so replaying the ledger cannot reconstruct `remaining` for adjustments without consulting `audit_logs` (which has the signed `difference_kg`). RECEIVED (+) and WASTE (−) are sign-unambiguous.

## 6. Failure-mode map

| Scenario | Current behaviour | Severity |
|---|---|---|
| Stale count applied after a waste/write-off | **Clobbers the waste** (sets batch to counted; intervening waste lost) | **I1 — High/P1** |
| Two staff applying counts at once (same line) | Safe — `FOR UPDATE` + `applied_at` idempotency | OK |
| Two staff, different sessions, same batch | Second apply clobbers first (stale `system_weight`) | I1 |
| Intake racing a count | New batch simply not in this count; no clobber | Low |
| Negative stock | Impossible (`CHECK >=0` + validation in waste/adjust) | OK |
| Missing batch | RPCs raise `Stock item not found` (P0002) | OK |
| Branch mismatch | `is_branch_manager(branch_id)` denies; RLS scopes reads | OK (audited via V12.4) |
| Partial failure after/before audit | None — single transaction; audit rolls back with the mutation | OK |
| Manual adjust over a concurrent change | Overwrites (no expected-value CAS), owner-only | I2 — Medium |

## 8. Recommended implementation plan (for approval — not started)

**Does CAS already exist?** Pessimistic locking: **yes**. Optimistic CAS / row_version: **no**. Immutable ledger: **yes** (`inventory_movements`). So V12.5 narrows to *adding a freshness guard at apply* (+ optional manual-adjust guard + ledger sign fix).

1. **CAS at stock-count apply (I1, primary).** In `ops_apply_stock_count_line`, under the batch `FOR UPDATE`, compare the batch's **current** `remaining_weight_kg` to the line's `system_weight_kg`. If they differ → the count is stale → **reject** with a clear "stock changed since you counted — please re-count" error (and surface in UI), rather than clobbering. (Equivalently: thread an `expected_remaining` into `admin_adjust_inventory_remaining`.)
2. **Expected-value CAS for manual adjust (I2, optional).** Add an `expected_remaining_kg` parameter to `admin_adjust_inventory_remaining`; reject on mismatch. UI passes the value it displayed.
3. **Ledger sign fix (I3, optional).** Record `ADJUSTMENT` movements as signed (or add a `direction`/signed `delta_kg` column) so the ledger is self-reconciling.
4. **row_version?** **Not required.** `system_weight_kg` is the natural baseline for the count path; an `expected_remaining` param covers manual adjust. A generic `row_version` on `inventory_batches` is a heavier alternative — only adopt if a uniform CAS across all paths is wanted.
5. **CAS location:** **SQL-side only**, inside the existing SECURITY DEFINER RPCs under the current lock. No app-layer concurrency logic; do not weaken existing behaviour.
6. **Migration needs:** one forward-only migration recreating `ops_apply_stock_count_line` (and optionally `admin_adjust_inventory_remaining` signature + a signed ledger column). Preserve V12.1/V12.4 grants & audit.
7. **UI conflict UX:** stock-count screen must handle the "stale count" rejection — show which batch drifted and offer re-count; do not silently swallow.
8. **Test harness:** new `scripts/verify-inventory-integrity.mjs` (concurrent apply-vs-waste → no lost update; stale count rejected; two-session apply; idempotency preserved; ledger replay reconciles). Unit tests for any pure helpers.
9. **Risk level:** **Low–Medium** — contained to 1–2 RPCs + a UI error path; the lock and ledger already exist.
10. **Stop/go:** **GO** for a narrow V12.5 focused on the apply-time CAS (I1) first, then optional I2/I3.

## 9. Findings table

| ID | Severity | Finding |
|----|----------|---------|
| I1 | **High (P1)** | Stock-count apply has no apply-time CAS; a stale count overwrites intervening waste/adjust (lost update). `system_weight_kg` is captured but never re-validated against current `remaining` at apply. |
| I2 | Medium | `admin_adjust_inventory_remaining` has no expected-value guard; concurrent manual adjust can overwrite (owner-only, single human action → lower likelihood). |
| I3 | Medium | `inventory_movements` ADJUSTMENT stores `abs(delta)`; sign lost ⇒ ledger not self-reconciling without `audit_logs`. |
| I4 | Medium (by-design?) | `products.is_available`/`stock_status` decoupled from batch `remaining`; storefront/checkout availability ≠ real stock truth; "unavailable" vs "zero stock" conflated. |
| I5 | Low | "Uncounted" batches aren't surfaced as a distinct truth state during a count. |
| OK | — | Locking, atomicity, non-negativity, branch authority, in-txn audit, append-only ledger, no sale-interleave, no demo stock — all sound. |

No P0: inventory cannot go negative, be forged, or be cross-branch mutated; the risk is **accuracy** (lost updates), which fits the operator's "inventory is closer to revenue/stock accuracy" framing.

## 10. Executive summary
Inventory's concurrency foundations are solid (pessimistic locks, atomic audited RPCs, append-only ledger, no negative stock, no checkout interleave). The one material concurrency defect is the **stale stock-count apply (I1)**: it trusts the record-time snapshot and clobbers any intervening waste/adjustment. Secondary items are a missing manual-adjust guard (I2), a sign-losing ADJUSTMENT ledger entry (I3), and availability/stock truth decoupling (I4).

## 11. P0 / P1 launch blockers
- **P1: I1 — stale stock-count apply (lost update).** Directly corrupts stock accuracy; must get an apply-time CAS.
- No P0. I2/I3/I4 are P2 hardening/clarity.

## 12. Recommended implementation order
I1 (apply-time CAS + UI stale-count UX + concurrency harness) → I2 (manual-adjust expected-value) → I3 (signed ledger) → optional I4 truth-state surfacing (likely a later "Operational Truth" phase, not V12.5).
