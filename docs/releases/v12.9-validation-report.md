## 3. Required Secrets (authoritative list)

Production fails closed without these (verified by the readiness script):
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `ORDER_ACCESS_SECRET` (>= 32 bytes)
- `STAFF_SESSION_SECRET` (>= 32 chars; falls back to `ORDER_ACCESS_SECRET`)
- `CANONICAL_BRANCH_ID` (or `NEXT_PUBLIC_CANONICAL_BRANCH_ID`)

Conditionally required: `TWILIO_*` + `SMS_SENDING_ENABLED` (only when SMS is on).
Must NOT be enabled in production: `ALLOW_DEMO_DATA`, `CHECKOUT_TEST_MODE_ENABLED`,
`NEXT_PUBLIC_CHECKOUT_TEST_MODE`, `MIGRATION_DRIFT_CHECK_MODE=local|dev`.

## 4. Readiness Verifier

`scripts/verify-production-readiness.mjs`:
- **STRICT** mode (`PRODUCTION_READINESS_MODE=strict`, `NODE_ENV=production`, or
  `VERCEL_ENV=production`): a missing required secret => FAIL.
- **LOCAL** mode (default dev): a missing production-only secret => SKIP, so the
  V12.10 certification gate is runnable locally. A PRESENT-but-INVALID secret
  (e.g. a too-short `ORDER_ACCESS_SECRET`) FAILS in either mode, and migration
  drift FAILS whenever the DB is reachable.

This mirrors the established skip-when-unconfigured pattern of
`check-migrations.mjs` and `verify-checkout-integrity.mjs`.

## 5. Migration Parity Finding (local dev artifact)

When the verifier was pointed at the **local dev DB**, parity reported version
`202606061200` (V12.1 RPC authority seal) as not recorded in
`supabase_migrations.schema_migrations` (the local DB has 22 recorded versions vs
20 repository files). This is a **local-dev migration-recording artifact, not a
repository or functional defect**:
- the repository migration file exists and the repo migration set is internally
  consistent;
- the V12.1 seal's *effects* are verified present — `verify-checkout-integrity.mjs`
  confirms anon is denied `create_checkout_order` (the V12.1 seal), and
  `get_migration_health()` reports all required migrations applied.

In LOCAL mode (no DB env) the parity check SKIPS, so it does not produce a false
failure in the certification gate. Production strict mode must be run against the
production DB after `supabase db push`, where parity must be clean.

## 9. Remaining Risks

- Strict secret validation is a **production-only step**: it cannot fully pass in a
  dev environment lacking the real secrets (it SKIPS there). It must be run with
  the production env before go-live.
- Backup/restore are operator-executed Supabase procedures; this repo documents
  and validates them but does not (and must not) exercise a real production
  restore.
- The local dev DB's `schema_migrations` recording is out of sync with the repo
  (§5); this does not affect production, where parity must be clean after push.
