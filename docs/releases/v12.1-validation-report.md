## Migration summary

New forward-only migration:

- `202606061200_v12_1_rpc_authority_seal.sql`

Migration actions:

- Drops obsolete public mutation/read overloads:
  - `create_checkout_order(uuid,text,text,text,date,uuid,text,text,jsonb)`
  - `cancel_public_order(uuid,text)`
  - `cancel_order_by_ref(text,text)`
  - `get_public_order(text)`
- Revokes direct client mutation grants from:
  - `create_checkout_order(..., jsonb, boolean)`
  - `check_rate_limit(text,text,integer,integer)`
  - `establish_public_order_access(text,text)`
  - `cancel_public_order(uuid,text,integer)`
  - `emit_audit_log(text,text,uuid,uuid,jsonb,text)`
- Grants those authority RPCs to `service_role`.
- Revokes client table mutation grants on `public_rate_limits`.
- Recreates `transition_order_status` as `SECURITY DEFINER` while preserving
  explicit `auth.uid()`, branch, status-transition, and audit checks.
- Adds a DDL event trigger that revokes client EXECUTE from newly created public
  functions, because default ACL revocation alone did not enforce the invariant
  in the local Supabase/Postgres bootstrap.
- Contains self-verification that raises if forbidden grants/functions remain.

## Findings corrected during validation

- Local dev database had direct `anon`/`authenticated` table mutation grants on
  `public_rate_limits`; the migration now explicitly revokes them before
  asserting.
- `ALTER DEFAULT PRIVILEGES ... REVOKE EXECUTE ... FROM PUBLIC` did not stop new
  public functions from being client-executable in the local bootstrap. V12.1 now
  enforces this with an event trigger and a live migration probe.
- Existing V11 static tests assumed `rate-limit.ts` could not use service-role
  transport. The test was updated to preserve containment while allowing the new
  server-mediated rate-limit authority.

## Unresolved risks

- Production migration parity is not proven from local tracked files; operator
  must run the release runbook against the linked Supabase production project.
- The new event trigger intentionally changes future migration ergonomics:
  client-callable RPCs must be explicitly granted after creation.
- Authenticated operational/admin RPCs remain broad by design and are protected
  by route middleware, server actions, RLS, and SQL branch/role checks. A later
  phase may choose to further narrow owner-only release RPC grants.
