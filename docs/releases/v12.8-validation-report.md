## 1. Executive Summary

V12.8 makes the system explain itself. It adds a unified structured logging layer,
in-process operational counters, explicit health states, a public `/api/health`
endpoint, a synthetic health verifier, and a vendor-neutral alert abstraction, then
wires minimal instrumentation at the clean chokepoints identified in discovery.

Silent failure is now visible: failed/locked logins, authority denials, checkout
faults (classified as `rpc_denied` vs `database_error`), stale stock-count
rejections, checklist completion failures, and audit-emit failures are all counted
and logged in a single redacted JSON format.

No hardened behaviour changed. All prior V12 verification gates remain green.

## 3. Structured Logging

`src/lib/server/observability/log.ts` is edge-safe (no `server-only`, no node
builtins) so it can be called from Edge middleware, server actions, and route
handlers — mirroring the `security-audit.ts` constraint.

Categories: `AUTH`, `CHECKOUT`, `AUDIT`, `INVENTORY`, `OPS_CAPTURE`, `RELEASE`,
`SYSTEM`. Every entry carries `timestamp`, `category`, `severity`, `message`, plus
`branchId` / `requestId` where relevant. Output is one JSON line per event via the
console sink keyed by severity.

Forbidden content is stripped defensively:
- secret-like keys (`pass*`, `token`, `secret`, `cookie`, `authorization`,
  `auth_header`, `service_role`, `api_key`, `bearer`, `credential`, `private_key`)
  are dropped;
- any value shaped like a JWT is redacted, even under an innocent key, recursively
  through nested objects/arrays.

## 4. Metrics

`src/lib/server/observability/metrics.ts` provides in-process counters:

`checkout_success`, `checkout_failure`, `checkout_partial_success`,
`login_success`, `login_failure`, `authority_denied`,
`inventory_stale_rejection`, `checklist_completion_failure`, `rpc_denied`,
`database_error`.

`noteRpcFault()` classifies an RPC error as `rpc_denied` (permission/RLS) or
`database_error` (anything else) and increments it.

Wiring summary:
- `checkout_success` / `checkout_failure` — `createCheckoutOrder`.
- `login_success` — `loginAction` success; `login_failure` / `authority_denied`
  — central `recordSecurityEvent` reason map.
- `inventory_stale_rejection` — `applyStockCountLine` on `STALE_STOCK_COUNT`.
- `checklist_completion_failure` — `completeChecklist` error.
- `rpc_denied` / `database_error` — checkout unknown faults + audit emit failure.
- `checkout_partial_success` — reserved; no order-created-then-secondary-failed
  path exists at create time today (documented in discovery §7).

Limitation (by design): counters are per-runtime-instance and reset on cold start.
Durable forwarding to an external metrics system is deferred.

## 5. Health States + Endpoint

`src/lib/domain/health.ts` defines `HealthState` = `HEALTHY` | `DEGRADED` |
`UNAVAILABLE` | `CONFIGURATION_REQUIRED`, with `worstState` folding and `isServing`.

`src/app/api/health/route.ts` (`GET /api/health`, dynamic) reports:
- `app` health,
- `configuration` (required secrets + canonical branch in production),
- `database` connectivity (anon `get_migration_health`),
- `migration_parity` (applied vs required required migrations),
- plus the metrics snapshot.

No secrets are exposed: the body carries states and generic, non-secret detail;
full DB error detail goes to the log. Status is 200 when serving (HEALTHY/DEGRADED)
and 503 when not (CONFIGURATION_REQUIRED/UNAVAILABLE).

## 6. Synthetic Checks

`scripts/verify-health.mjs` verifies, against the DB (always when reachable):
- database connectivity (`get_migration_health`),
- migration parity (all required applied),
- checkout readiness (service-role DB reachable),
- checkout RPC seal (anon denied — V12.1).

When `HEALTH_BASE_URL` is set it additionally checks the home page, shop page, and
the `/api/health` endpoint shape. It SKIPS (does not fail) when no app/DB is
reachable, mirroring `verify-checkout-integrity`.

## 7. Alert Readiness

`src/lib/server/observability/alerts.ts` is a vendor-neutral abstraction:
- console sink always (redacted via the structured logger),
- webhook sink when `ALERT_WEBHOOK_URL` is set (minimal taxonomy payload, no
  context fields on the wire),
- a Sentry sink stub for a future phase.

No production provider is required or hardcoded.

## 9. Remaining Risks

- In-process counters are per-instance and reset on cold start; durable metric
  forwarding (and a production alert provider such as Sentry) is deferred.
- `checkout_partial_success` is defined but currently unwired — no create-time
  partial-success path exists; it is reserved for a future secondary-step case.
- Lower-risk intelligence loaders still use ad-hoc `console.*`; they can adopt the
  structured logger incrementally (out of V12.8 scope).
- HTTP-layer health/checkout checks were skipped locally (no running app server);
  DB-layer checks passed.
