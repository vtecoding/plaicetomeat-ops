## 1. Executive Summary

V12.6 makes opening/closing checklists evidence-backed at the database boundary. Checklist sessions now bind to versioned server-known definitions, step keys are validated in SQL, evidence payloads are type/bounds checked, and completion is derived from persisted required step events before completion state or audit evidence is written.

Closed gaps:
- unknown step keys are rejected
- empty checklists cannot complete
- incomplete checklists cannot complete
- number evidence must be `{ value: number }`
- temperature evidence is bounded
- confirmation/skipped steps cannot carry arbitrary evidence payloads
- completion audit emits only after valid completion
- receipts can carry session/definition/actor/branch/completion metadata while legacy receipts remain readable

## 6. Temperature / Compliance Coupling

Implemented in V12.6:
- guided checklist temperature evidence is strictly typed and bounded.
- opening `fridge_temp` and closing `fridges_closed` require numeric `{ value }` payloads when marked done.

Deferred:
- full single-source merge with the older compliance temperature model.

Reason:
- the older `/counter/compliance` path is demo-backed and not a live canonical persistence flow.
- merging it with guided checklist evidence is a broader compliance-domain design, already called out as deferred in prior V11 notes.
- V12.6 does not claim duplicate temperature/compliance models are fully unified; it only prevents invalid guided-checklist evidence from being accepted as valid.
