## 1. Executive summary

V12.3 was a **containment/hardening slice, not a checkout rewrite** — the money
path's authority and atomicity were already sound (V12.1 + prior). This release
closes the abuse-resistance and concurrency gaps from discovery:

- **One hardened checkout service.** Both the storefront action and `POST /api/checkout`
  go through `submitCheckout`, which now does payload caps → duplicate-SKU merge →
  schema validation → rate limit → server-only test gate → service-role RPC. There is
  no second, looser mutation path.
- **`/api/checkout` retained and hardened (not removed).** It is used by the e2e suite
  (`tests/e2e/checkout.spec.ts`), so per the "prove use before deleting" guardrail it was
  hardened: same body cap (413), same hardened service, same throttle and caps.
- **Rate limiting before mutation** using the existing `checkout` bucket (12/300s),
  fail-closed, keyed on the hashed network identity.
- **Pickup-window capacity is now race-free**: capacity is rechecked while holding a
  `FOR UPDATE` row lock on the window (the same deterministic-lock pattern
  `transition_order_status` uses) — concurrent checkouts create **exactly** `max_orders`,
  never more.
- **Duplicate SKUs merge before validation** (app layer *and* the RPC), so a duplicate-line
  payload can no longer bypass per-product max quantity.
- **Payload caps**: ≤ 30 distinct SKUs (after merge), ≤ 32 KiB body, bounded contact /
  notes / idempotency-key fields.
- **Idempotency tightened**: same key + identical payload returns the one order; same key +
  **different** payload is now **rejected** (fingerprint), instead of silently returning the
  original.
- **Fault isolation**: a committed order never reports total failure if the access cookie
  can't be set — it returns a typed partial success (`orderPlaced: true,
  accessEstablished: false, recoveryRequired: true`); and a missing `ORDER_ACCESS_SECRET`
  is **preflighted before** any mutation.
- **V12.1 authority seal preserved**: `create_checkout_order` remains `service_role`-only;
  anon/authenticated cannot call it (re-asserted + self-checked in the migration).

## 3. Checkout architecture after V12.3

```
Storefront action (createOrderAction)              POST /api/checkout
  ├─ preflight ORDER_ACCESS_SECRET (refuse pre-mutation)   ├─ 32 KiB body cap → 413
  ├─ 32 KiB basket cap                                     ├─ JSON parse → 400
  └────────────────────────┬──────────────────────────────┘
                           ▼
              submitCheckout()  ── the single hardened service
                 1. merge duplicate SKUs (prepareCheckoutInput)
                 2. zod validate (caps: ≤30 SKUs, ≤20/SKU, bounded fields)
                 3. service env present?
                 4. checkRateLimit("checkout", networkHash, failClosed) → 429
                 5. server-only test-order gate (CHECKOUT_TEST_MODE_ENABLED)
                 6. create_checkout_order (service_role RPC)
                           ▼
   RPC (atomic txn): merge → fingerprint idempotency → branch active →
   LOCK window FOR UPDATE → recheck capacity → recompute price/subtotal →
   per-product max on merged qty → insert order/items/status/audit
                           ▼
   action only: grantOrderAccess (best-effort) → full | partial success
```

## 4. Security changes
- Public order creation is **throttled before mutation** on every path (fail-closed).
- `/api/checkout` is no longer an unbounded mutation path: body cap + identical hardened service + same throttle/caps.
- **Forged client price is impossible** (price is not an RPC parameter; line totals/subtotal are recomputed from `products.price_per_unit`) — proven for both the RPC and the HTTP endpoint.
- **Test-order flag** remains server-env gated.
- **V12.1 seal preserved**: anon/authenticated cannot execute `create_checkout_order`; the migration re-asserts the grant and self-checks it.
- No new anon/authenticated mutation grants introduced.

## 5. Database / migration changes
- Forward-only `202606071000_v12_3_checkout_integrity.sql` (historical migrations untouched; production order preserved).
- `ALTER TABLE orders ADD COLUMN IF NOT EXISTS idempotency_fingerprint text`.
- `CREATE OR REPLACE FUNCTION create_checkout_order(...)` — **same signature** (so the seal/grants are preserved), now with: SKU merge, distinct-SKU cap (30), capacity recheck under `FOR UPDATE`, fingerprint-based idempotency (same key + different payload rejected; legacy NULL fingerprints tolerated).
- Self-verifying: a `DO` block asserts the column exists and that anon/authenticated cannot execute the function (the migration fails if the seal is broken).
- Applied to the local stack and registered in `supabase_migrations.schema_migrations`.

## 6. Concurrency proof
`verify-checkout-integrity.mjs` fires **8 concurrent** `create_checkout_order` calls into a dedicated `max_orders = 3` window:
- **committed orders == 3 exactly** (no overbooking), **5 rejected with “full”**.
- Idempotent retry (same key) → same order ref and **exactly one** order row.
- Same key + different payload → **rejected** (“different details”).
The guarantee comes from the `FOR UPDATE` window-row lock: the capacity count→insert is serialized per window for the transaction's duration.

## 7. Abuse-resistance proof
From the harness (DB + HTTP):
- Duplicate SKUs **merge** to one line with summed quantity; merged quantity over per-product max is **rejected** (no dup-line bypass).
- **> 30 distinct SKUs rejected** (RPC + schema).
- `/api/checkout` **413** on > 32 KiB body; **400** on malformed JSON.
- `/api/checkout` **throttles (429)** via the checkout bucket; created ≤ 12 (bucket max) — throttle stops mutation.
- Forged client price **ignored** (stored line price == DB price) on both RPC and HTTP paths.

## 9. Remaining risks
- **Throttle thresholds** (12/300s, fail-closed) are conservative for a single shop; tune if real customers are blocked (e.g. a family ordering from one network). Currently invalid payloads fail validation *before* consuming budget, which limits false lockouts.
- **Window-row lock** serializes checkouts per window; fine at shop scale, but a very hot single window would queue briefly under load.
- **Legacy idempotency rows** (pre-migration, NULL fingerprint) are tolerated on retry (cannot detect payload drift for those) — affects only orders placed before this migration.
- **`/api/checkout`** has no API key; it is rate-limited and fully server-authoritative, but if it's meant as a real external API it should get a key/allowlist in a later slice (out of V12.3 scope).
- **Stock is still not reserved at checkout** (pay-on-collection, cut-to-order) — unchanged design, documented in discovery; not a V12.3 concern.
- Production cutover requires applying the forward migration and setting secrets (operator step).
