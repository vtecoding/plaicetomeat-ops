## 1. Executive summary

Discovery found the audit system's **authenticity sound but its coverage of
security/authority failures absent** ‚Äî `security_event` was defined but never
produced, and the V12.2 identity/authority surface emitted nothing. V12.4 fixes
exactly that, without touching the authoritative in-transaction business-RPC audit
path.

- **`security_event` is now a live, canonical event type** with 11 stable reason codes.
- **A real, service-authoritative emitter** (`recordSecurityEvent`) replaces the dead
  `emitSecurityEvent`. It is edge-safe (works in the Edge middleware *and* Node), and
  routes through the **hardened, service_role-only `emit_audit_log`** ‚Äî so V12.1's RPC
  authority seal is fully preserved (no authenticated/anon emission reintroduced).
- **Every V12.2 security/authority refusal now emits evidence**: failed login, lockout,
  missing/expired/tampered/cross-user session, role denial, null-branch denial,
  cross-branch denial, owner-only denial, and unauthorised-route hits. Logout failure
  is audited too.
- **Payloads are PII-free**: hashed email/network, reason codes, route, role, branch_id,
  user id (uuid) ‚Äî never raw email, IP, token, cookie, or profile. `emit_audit_log`
  also strips secret-like keys defensively (verified end-to-end).
- **Audit-door consolidation is explicitly deferred** (see ¬ß6): the 30 inline business
  inserts are already authoritative, in-transaction, and unforgeable, so rewiring them
  is out of scope and unnecessary for coverage.

Gate: typecheck ‚úì, lint ‚úì (4 warnings, 0 errors), **356 unit tests** ‚úì (was 338),
build ‚úì (middleware bundle compiles edge-safe), and **all five** verify scripts green
incl. the new `verify-audit-coverage.mjs`.

## 4. Security-event payload schema

`emit_audit_log(event_type='security_event', target_type, target_id, branch_id, metadata, system_reason)`:

- **system_reason**: one of the 11 reason codes (recorded as `metadata.system_reason`; system emission ‚áí `actor_id = NULL`).
- **target_type**: `"auth"` | `"session"` | `"authority"`.
- **target_id**: the staff user id (uuid) when known, else null. *(A uuid, not PII.)*
- **branch_id**: where safe (e.g. target branch on cross-branch denial), else null.
- **metadata** (safe fields only):
  - auth: `{ emailHash, networkHash, detail? }` ‚Äî **hashed**, never raw.
  - session: `{ route, detail? }`.
  - authority: `{ role, requiredRole?, ownBranchId?, route?, active? }`.

**Never present:** raw email, raw IP, session token, auth cookie value, full profile.
Defence in depth: `emit_audit_log` strips secret-like keys (`token`, `secret`,
`cookie`, `jwt`, `session`, ‚Ä¶) and caps metadata size; proven by
`verify-audit-coverage.mjs` (a planted `token` key is redacted into `_redacted_keys`).

## 6. Deferred audit-door consolidation note

The discovery report identified **two emission doors**: the hardened
`emit_audit_log` (allowlist + secret-strip + size cap) and ~30 inline
`INSERT INTO public.audit_logs` statements inside business RPCs.

**Consolidation is deliberately deferred.** The inline business-RPC inserts are:
- **authoritative** ‚Äî written by trusted `SECURITY DEFINER` functions in the table-owner context;
- **in-transaction** ‚Äî atomic with the mutation they record (discovery Q6: none are out-of-transaction);
- **unforgeable** ‚Äî direct client writes are revoked (V11.2) and the RPCs are sealed (V12.1).

Rewiring 30 correct, atomic inserts to route through `emit_audit_log` would add risk
(breaking atomicity if done as a nested call, or churning 18 migrations) for no
integrity gain. V12.4 instead made the *missing* path real and used it for the
security/authority surface. A future hardening slice may add a DB-level assertion
(trigger) that rejects off-allowlist `event_type` / secret metadata on `audit_logs`
directly ‚Äî captured as a candidate, not done here.

## 7. Remaining risks
- **Middleware emission is best-effort** (`event.waitUntil`, fire-and-forget): under an edge-platform kill a session event could be dropped. Acceptable for security telemetry; the authoritative business audit is unaffected.
- **Volume**: repeated failed logins / denials produce one event each. Throttling/aggregation is not implemented; for a single shop this is fine, but a brute-force burst would write many rows. (Login lockout already bounds repeated attempts.)
- **A2 (two doors) remains** by design (see ¬ß6) ‚Äî inline inserts are not subject to `emit_audit_log` hygiene; mitigated by their being trusted, reviewed code.
- **No new migration** ‚áí nothing to apply for V12.4; production cutover is code-only (set secrets already covered by V12.2/V12.3).
- Middleware now imports `@supabase/supabase-js` (edge) for the emitter ‚Äî bundle grew modestly (91.5 kB); compiles and runs.
