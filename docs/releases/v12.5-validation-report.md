## 1. Executive Summary

V12.5 seals the stock-count lost-update defect with a SQL-side hard rejection. `ops_apply_stock_count_line` now locks the count line, validates the session/branch, locks the inventory batch row, and compares the current `inventory_batches.remaining_weight_kg` to the count line's recorded `system_weight_kg` before any apply mutation.

If the batch changed after counting, the RPC raises:

`STALE_STOCK_COUNT: stock changed since this count was recorded; re-count before applying.`

On stale rejection, the function does not:
- mutate `inventory_batches`
- mark the count line applied
- write an `ADJUSTMENT` correction movement
- emit `stock_count_line_applied` success audit

Existing pessimistic locking, branch authority, negative-stock protection, idempotent re-apply behavior, and stock-count happy paths remain intact.

## 5. Deferred Items

Manual-adjust expected-value guard:
- Deferred deliberately.
- Correct implementation would need changing or removing the existing three-argument `admin_adjust_inventory_remaining` RPC so direct callers cannot bypass the expected value.
- That is a wider RPC/signature contract change touching app actions and existing verification scripts.
- V12.5 still blocks stale stock-count applies, including after manual adjustment, which was the launch-blocking lost-update defect.

Ledger sign issue:
- Deferred deliberately.
- Existing `inventory_movements.quantity_kg` has a positive-only semantic (`CHECK > 0`) and historical `ADJUSTMENT` rows store `abs(delta)`.
- V12.5 does not claim the ledger is replayable from `quantity_kg` alone.
- A future forward migration should add an explicit signed delta field or otherwise preserve signed movement semantics without breaking historical consumers.
