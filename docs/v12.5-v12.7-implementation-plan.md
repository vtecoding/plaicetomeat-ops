## 4. Migration Targets

### V12.5

Create a forward-only migration:
- `202606071600_v12_5_inventory_stale_count_guard.sql`

Required SQL changes:
- `CREATE OR REPLACE FUNCTION public.ops_apply_stock_count_line(uuid, uuid, text)`
- Lock `stock_count_lines` row with `FOR UPDATE` as today.
- Lock the matching `inventory_batches` row with `FOR UPDATE` before any apply decision.
- Verify line/session/batch branch consistency.
- Compare current `inventory_batches.remaining_weight_kg` to `stock_count_lines.system_weight_kg`.
- If different, raise typed stale-count failure:
  - message: `STALE_STOCK_COUNT: stock changed since this count was recorded; re-count before applying.`
  - SQLSTATE: use a stable non-success code such as `P0001` unless a more precise project pattern exists.
- On stale failure:
  - do not update `inventory_batches`
  - do not update `stock_count_lines.applied_at`
  - do not insert `inventory_movements`
  - do not insert success audit rows
- Preserve idempotent re-apply for already applied lines.
- Preserve existing `GRANT EXECUTE ... TO authenticated`.

Risk-gated optional SQL changes:
- Manual adjustment expected-value guard:
  - preferred implementation is a new overload or optional fourth argument for `admin_adjust_inventory_remaining` with `p_expected_remaining_kg`.
  - only implement if app and existing script call sites can be updated safely without broad UI churn.
  - otherwise document deferred in V12.5 validation.
- Ledger sign issue:
  - current `inventory_movements.quantity_kg` has `CHECK > 0`, so signed deltas cannot be stored in-place without changing table semantics.
  - low-risk option is to add nullable `delta_kg numeric` for ADJUSTMENT rows and populate it for new adjustments.
  - if schema/reporting compatibility is uncertain, document deferred explicitly rather than claiming the ledger is replayable.

## 5. Risk Map

### V12.5

- High risk: stale apply fix could accidentally emit success audit or movement before rejection. Mitigation: SQL guard before mutation and verification assertions for no movement/no audit.
- Medium risk: changing `admin_adjust_inventory_remaining` signature could break existing actions/scripts. Mitigation: keep old signature or defer expected-value guard if too wide.
- Medium risk: signed ledger change conflicts with positive `quantity_kg` semantics. Mitigation: add a new signed column only if safe; otherwise document deferred.
- Low risk: stale-count error may be hidden by `safeMessage`. Mitigation: add stale error text to safe patterns.

### V12.6

- High risk: server-side checklist definitions may diverge from TypeScript UI definitions. Mitigation: centralise version/key manifest or generate SQL data from a small fixed contract.
- High risk: completion rules could break historical receipts. Mitigation: keep old receipts readable and enforce new completion only on active/new sessions.
- Medium risk: temperature evidence validation could reject legitimate historical payload shapes. Mitigation: apply strict validation on writes; reads tolerate legacy rows.
- Medium risk: duplicate temperature/compliance paths may need a larger design. Mitigation: document any larger dedupe work as deferred if it exceeds the narrow integrity fix.

### V12.7

- High risk: removing demo fallback can make production pages visibly unavailable. This is intended when data/config is missing, but UI must be honest and usable.
- High risk: changing `getPublicBranch` semantics affects storefront, checkout, SMS, header, and settings display. Mitigation: add explicit canonical branch result/state and update callers deliberately.
- Medium risk: DataResult propagation can become a broad refactor. Mitigation: start with loaders feeding `/admin`, `/admin/today`, Business Insights, releases, catalog/inventory/order reads.
- Medium risk: empty arrays currently mean both no data and failed data. Mitigation: use typed states at boundaries before changing UI copy.

## 7. Rollback Notes

- No historical migration edits. Rollback is forward-only.
- If V12.5 stale guard causes operational friction, add a future forward migration to create an explicit re-count/reopen workflow. Do not remove the stale guard.
- If manual-adjust expected-value guard is implemented and proves too disruptive, add a forward migration that restores backward-compatible overload behavior while preserving stale-count protection.
- If V12.6 checklist definitions need correction, add a new checklist definition version rather than rewriting historical sessions.
- If V12.7 canonical branch configuration is missing in production, configure the canonical branch; do not restore first-active/demo fallback.
- Code rollback may revert app-layer changes before deployment, but once migrations are applied, subsequent corrections must be forward migrations.
