# V12.3 Discovery Report — Checkout Integrity

Date: 2026-06-07
Scope: V12.3 (Checkout Integrity) **discovery only**. No implementation, no migrations, no test edits.
Builds on: V12.1 (DB authority seal) + V12.2 (identity/branch authority, committed `39428c1`).
Out of scope (do not start): audit semantics, inventory concurrency, observability, pricing provenance, production certification.

This maps the money path from browser submit → server action → validation →
(missing) rate-limit → RPC mutation → order-access cookie → confirmation/recovery,
with evidence. **Where reality differs from the V12 master spec, reality is
documented with evidence and wins.**

---

## 1. Executive summary

The checkout path is **stronger than expected on authority and atomicity, weaker
than expected on abuse-resistance and concurrency.**

Already solid (verified):
- **Price authority is server-side.** The client's `unitPriceSnapshot` is validated
  but **dropped before the RPC** — only `{productId, quantity}` is sent
  ([orders.ts:263](src/lib/server/orders.ts:263)); the RPC recomputes subtotal and
  line totals from `products.price_per_unit`
  ([202605310001:...create_checkout_order]). A forged client price cannot move money.
- **Idempotency is DB-enforced.** `orders.idempotency_key text UNIQUE NOT NULL`
  ([init:122](supabase/migrations/202605290001_init.sql:122)); the RPC returns the
  existing ref on key match and on `unique_violation`. The client key is generated
  once per page and stable across retries ([checkout-client.tsx:72](src/components/checkout-client.tsx:72)).
- **One atomic transaction.** Order + items + status event + audit are written in a
  single SECURITY DEFINER plpgsql function — all-or-nothing.
- **Test-order flag is server-gated.** `isTest` is honoured only when
  `CHECKOUT_TEST_MODE_ENABLED=true` server-side ([orders.ts:99](src/lib/server/orders.ts:99)); a forged client flag is ignored in prod.
- **Direct RPC is sealed (V12.1).** `create_checkout_order` is `service_role` only;
  anon/authenticated were revoked ([v12.1:70-75](supabase/migrations/202606061200_v12_1_rpc_authority_seal.sql:70)).

Gaps (the V12.3 work):
- **No rate limiting on order creation.** A `checkout` bucket exists in `RATE_LIMITS`
  but is **never called** by the action, `submitCheckout`, or the API route.
- **Pickup-window capacity is a check-then-insert race** — `count(*) … >= max_orders`
  then `INSERT`, no lock, READ COMMITTED — so concurrent orders can overfill a window.
- **No basket size / SKU-count cap** and **duplicate SKUs aren't aggregated**, so the
  per-SKU max is bypassable and oversized payloads are accepted.
- **`/api/checkout` is an unauthenticated, uncapped public order-creation endpoint.**
- **Order-success vs cookie-establishment is not fault-isolated**: if the cookie step
  throws (e.g. missing `ORDER_ACCESS_SECRET`), the *created* order is reported to the
  user as a failure.

No P0 lets an attacker steal money or read others' orders; the P0s are **order spam
/ DoS and window overbooking**.

## 3. Current checkout architecture

### 3.1 Checkout flow map
1. **Page** [`/checkout`](src/app/checkout/page.tsx) (server) — resolves `branch = getPublicBranch()`, active pickup windows, branch settings (min order, cutoff). Public, no auth (correct — customer-facing).
2. **Client** [`checkout-client.tsx`](src/components/checkout-client.tsx) — basket from `localStorage`; generates `idempotencyKey` once; mirrors phone/min-order/cutoff validation for UX; posts a server-action `FormData` (basket JSON, branchId, idempotencyKey, isTest, customer fields).
3. **Server action** [`createOrderAction`](src/app/actions/checkout.ts) — parses basket JSON, calls `submitCheckout`, then `grantOrderAccess(publicAccessId, 1)` to set the signed HttpOnly cookie; returns `{orderRef, publicAccessId}`.
4. **Validation** [`submitCheckout`](src/lib/server/orders.ts:78) — `createCheckoutSchema` (zod) re-validates everything; gates `isTest` on server env; requires service env.
5. **Rate-limit layer** — **absent** in this path (C1).
6. **RPC/mutation** [`createCheckoutOrder`](src/lib/server/orders.ts:215) → `create_checkout_order` via **service-role** client; maps basket to `{productId, quantity}` only.
7. **Cookie/order-access** [`grantOrderAccess`](src/lib/server/order-access-session.ts:53) — HMAC-signed `ptm_order_access` grant, scoped to `/order`.
8. **Confirmation/recovery** — client redirects to `/order/status/{publicAccessId}`; lost-cookie recovery via `/order/lookup` (ref + phone → `establish_public_order_access`).
- **Alt entry**: [`POST /api/checkout`](src/app/api/checkout/route.ts) → `submitCheckout` (service role), returns `publicAccessId` but sets **no** cookie. Unauthenticated, uncapped (C6).

### 3.2 RPC body (authority-bearing logic)
`create_checkout_order` (canonical body [202605310001](supabase/migrations/202605310001_v2_phase_e_sms_test_mode.sql); jsonb-returning re-decl in [v11.1](supabase/migrations/202606051200_v11_1_public_order_access.sql); sealed to service_role in [v12.1](supabase/migrations/202606061200_v12_1_rpc_authority_seal.sql)):
- idempotency short-circuit → branch active → window belongs to branch+active+day-of-week → past-date / same-day-cutoff → shop-closure → **capacity count** → product availability/min/max → min-order-value → `next_order_ref` → INSERT order/items/status-event/audit → return `{orderRef, publicAccessId}`.

## 4. Authority map

| Field | Origin | Trust |
|---|---|---|
| `unitPriceSnapshot` / subtotal | client (display) | **untrusted; dropped before RPC**; recomputed from `products.price_per_unit` server-side |
| `productId`, `quantity` | client | passed to RPC; validated against `products` (availability, min/max qty) |
| `unitType`, product name | client (display only) | RPC uses `products.unit_type` / `products.name` (snapshot) — server-authoritative |
| `branchId` | client hidden field (from `getPublicBranch()`) | validated active in RPC; single-branch storefront |
| `pickupWindowId` | client | RPC enforces window∈branch, active, day-of-week, capacity |
| `pickupDate` | client | zod + RPC (past/cutoff/closure) in **branch timezone** |
| `customerName/phone/email/notes` | client | zod (UK mobile normalised, lengths); trimmed in RPC |
| `idempotencyKey` | client (uuid) | zod min-12; DB UNIQUE |
| `isTest` | client | **ignored unless** `CHECKOUT_TEST_MODE_ENABLED` server env |
| price/line_total/order_ref/subtotal | **server/DB only** | recomputed/generated |

Branch/window/product/price authority all resolve **server-side in SQL**. This is the key strength.

## 5. Mutation map
- **RPC**: `create_checkout_order` (SECURITY DEFINER, `search_path=public`, service_role only).
- **Tables mutated**: `orders` (INSERT), `order_items` (INSERT), `order_status_events` (INSERT), `audit_logs` (INSERT). Reads: `branches`, `pickup_windows`, `branch_settings`, `shop_closures`, `products`, `orders` (count + idempotency).
- **Transaction boundary**: the entire function = one implicit transaction → atomic.
- **Audit/events**: `audit_logs(event_type='order_created', metadata{order_ref,subtotal,is_test})` + `order_status_events('incoming')`, inside the same txn.
- **Idempotency**: up-front SELECT by key + `unique_violation` handler → exactly-once per key (replay-safe).

## 6. Failure-mode map
- **Order insert OK but cookie/token establishment fails** → order is **committed**; `grantOrderAccess` runs in the action *after* the RPC. A throw there makes the action report failure though the order exists (C5). Recoverable: stable key → retry re-establishes; or `/order/lookup`. The user isn't shown their ref on this failure.
- **Checkout secret missing** → `ORDER_ACCESS_SECRET` unset in prod makes `getSecret()` throw inside `grantOrderAccess` ([order-access-session.ts:31](src/lib/server/order-access-session.ts:31)) — *after* the order commits → C5 manifestation. (Order creation itself doesn't need the secret.)
- **Pickup window full** → RPC raises `This pickup window is full.` (safe message) when `max_orders` set and count reached — **but only single-threaded** (see C2).
- **Duplicate SKU payload** → two array rows for one product: subtotal sums both, two `order_items` rows, per-SKU max bypassed (C4).
- **Concurrent checkouts (same window)** → capacity race overfills (C2). **(same idempotency key)** → unique constraint → one order, the loser returns the existing ref (safe).
- **Retry/replay** → idempotency-safe (DB unique).
- **Forged client price/subtotal** → no effect (price dropped; server recomputes).
- **Forged `isTest`** → ignored unless server env enables it.

## 7. Concurrency-risk map
- **Pickup-window capacity**: check-then-insert, **no `FOR UPDATE`, no advisory lock**, default READ COMMITTED → raceable overbooking (C2). Contrast: `transition_order_status`/`cancel_public_order` *do* use `SELECT … FOR UPDATE` ([v12.1:130](supabase/migrations/202606061200_v12_1_rpc_authority_seal.sql:130)) — the lock pattern exists, just not in checkout.
- **Stock/product quantity**: only availability/stock_status flags + min/max per line; **no stock reservation/decrement** at checkout (pay-on-collection, cut-to-order). Not oversold because stock isn't committed here — design reality, not a bug, but document.
- **Advisory locks**: none in checkout.
- **Isolation assumptions**: relies on READ COMMITTED; capacity invariant is not serializable.
- **Raceable paths**: window capacity (real); idempotency (covered); order_ref generation (`next_order_ref` — assess for its own locking in implementation).

## 8. Abuse-resistance map
- **Payload size**: no domain cap on basket length (C3); only Next.js's default body limit + per-SKU qty ≤ 20.
- **SKU count**: uncapped; duplicates allowed (C3/C4).
- **Rate limiting**: none on checkout (C1); `/api/checkout` fully open (C6).
- **Idempotency key**: client uuid, DB-unique — replay-safe, but not an abuse control (attacker varies the key).
- **Replay protection**: idempotency covers honest replays; not malicious volume.
- **V12.1 direct-RPC denial**: intact — anon/authenticated cannot call `create_checkout_order`; only the server service-role transport can.

## 9. Test inventory
- **Existing**: [`checkout.test.ts`](src/lib/validation/checkout.test.ts) — zod schema only (phone/date/cutoff). DB adversarial scripts exist for public-access/audit/ops but **none for checkout**.
- **Safe-test-order**: server gate logic (`isCheckoutTestModeEnabled`) untested directly.
- **Missing adversarial**: forged-price no-op; duplicate-SKU; oversized basket; rate-limit trip; capacity race (concurrent inserts); idempotency race; cookie-failure-after-commit; `/api/checkout` open-endpoint behaviour.
- **Required harness**: a DB integration/concurrency script (à la `verify-public-access.mjs`) firing concurrent `create_checkout_order` calls against a `max_orders=N` window to prove the post-fix invariant.

## 10. Recommended V12.3 implementation plan (pending approval — not started)

**Migrations needed**
- M1 (C2): make capacity atomic — `pg_advisory_xact_lock(hashtext(window||date))` (or a partial unique/serializable strategy) around the count→insert in `create_checkout_order`. Add the concurrency harness.
- M2 (C4): aggregate `p_items` by `product_id` (sum quantities) in the RPC and enforce per-SKU max on the aggregate; reject or merge duplicates.
- (No schema change strictly required for C1/C3/C5/C6 — app-layer.)

**Code targets**
- C1/C6: wire `checkRateLimit("checkout", clientNetworkHash(), { failClosed:true })` into `submitCheckout` (covers both the action and `/api/checkout`); map a trip to a generic "try again shortly".
- C3: add `.max(...)` to the basket array + a total-SKU cap in `createCheckoutSchema`.
- C5: fault-isolate — treat order creation as the success boundary; if cookie establishment fails, still return `{ok:true, orderRef}` and let the status page/lookup recover; log the cookie failure.
- C7/C8: minor — regenerate idempotency key when basket changes post-failure (optional); ensure the literal preview key can't reach submit.

**Test targets**
- New `scripts/verify-checkout-integrity.mjs` (live-DB): forged price no-op, duplicate-SKU aggregation, capacity race (N concurrent → exactly `max_orders` non-cancelled), idempotency race, rate-limit trip, `/api/checkout` throttled, test-flag gating.
- Unit: schema max/dup cases; `submitCheckout` env/test gating; cookie-failure fault isolation.

**Sequencing**: C1+C6 (throttle) → C2 (capacity lock + harness) → C4 (dedup) → C3 (caps) → C5 (fault isolation) → C7/C8 polish → validation report.

**Risks**: advisory-lock key choice (contention vs correctness); changing dup-SKU behaviour may affect existing baskets/UX; rate-limit thresholds for a single shop (avoid blocking real customers — `failClosed` only on the establishment-style buckets, conservative limits).

**Stop/go**: **GO** to implementation after approval; medium effort, mostly contained to `create_checkout_order` + `submitCheckout`, leveraging existing lock and rate-limit patterns.
