# V12.7 Discovery - Operational Truth Layer

Date: 2026-06-07
Scope: V12.7 discovery only. No implementation, no migrations, no test edits.
Builds on: V12.1-V12.6.

## 0. Verdict

V12.2 removed branch fallbacks from authenticated admin authority, but production data truth still degrades in several loader paths. Public catalog loaders can still return demo branch/products/categories/settings when Supabase is missing or a query fails. Counter/admin analytical loaders still collapse unavailable data into empty arrays or zero-valued summaries. Releases can show a hardcoded V3 fallback release when no release rows/config are available.

V12.7 should add a small typed truth layer and remove silent production demo fallback from production-facing loaders. The narrow implementation should focus on:
- typed `DataResult` states
- explicit canonical storefront branch resolution
- no production demo fallback for catalog/products/settings/pickup/order/admin intelligence/releases
- shared operational snapshot for `/admin` and `/admin/today`
- honest UI states for configuration missing, no data, unavailable, and degraded data

## 1. Demo Fallback Paths

### `src/lib/server/catalog.ts`

Current production risk:
- `getPublicBranch()` returns `demoBranch` when service env is missing or no active branch is found.
- `getBranchSettings()` returns `demoBranchSettings` when env/query/row missing.
- `getActiveCategories()` and `getAllCategories()` return `demoCategories` on env/query failure.
- `getPublicProducts()` and `getAllProducts()` return `demoProducts` on env/query failure.

This is the largest production demo fallback surface because it feeds:
- `/`
- `/shop`
- `/product/[slug]`
- `/checkout`
- `/our-halal-promise`
- settings display via `getPublicBranch`
- `site-header`
- SMS address lookup through `getPublicBranch`

### `src/lib/server/pickup-windows.ts`

Current production risk:
- `getPickupWindows()` returns `demoPickupWindows` on env/query failure.
- `getShopClosures()` returns demo closures when env missing, and empty on query failure.

### `src/lib/server/orders.ts`

Current production risk:
- `getCounterOrders()` returns `getDemoOrders()` when env missing or query fails.
- `getOrderById()` can return a demo order when env missing.

### `src/lib/server/operations-intelligence.ts`

Current production risk:
- if service env is missing, `buildFallbackIntelligence()` uses `getDemoOrders()` and returns an owner-facing intelligence object.
- data query errors are logged but the returned object still carries real-looking zero/partial values with `dataState.status = "error"`.

### `src/lib/server/releases.ts`

Current production risk:
- if env missing, returns `[fallbackV3Release()]`.
- if release rows are empty, returns `[fallbackV3Release()]`.

### Demo data that is explicit/static rather than a fallback

- `src/lib/data/demo.ts` remains useful for dev/test.
- `components/basket-client.tsx` seeds browser basket branch id from `demoBranch`.
- `components/countdown-banner.tsx` uses demo pickup windows.
- `src/app/counter/compliance/page.tsx` renders demo compliance readings. This is a separate operational-truth concern; it should not pretend to be live compliance evidence.

## 2. Public Branch / Canonical Branch Fallbacks

`getPublicBranch()` currently selects first active branch ordered by `created_at`, then falls back to `demoBranch`.

This is acceptable only as an explicit dev/demo mode. In production, storefront branch must be explicitly canonical. If no canonical branch is configured, loaders should return `CONFIGURATION_REQUIRED`.

Recommended implementation:
- use an environment variable `NEXT_PUBLIC_CANONICAL_BRANCH_ID` or `CANONICAL_BRANCH_ID` first, because it avoids a migration in this narrow slice.
- verify the branch exists and is active.
- if production has no canonical id, return `CONFIGURATION_REQUIRED`.
- keep first-active branch only for non-production development convenience if no canonical id is set.

## 3. Admin Dashboard Data Loaders

`/admin` uses:
- `getDashboardMetrics`
- `getOperationsIntelligence`
- `getShopIntelligence`

Current truth issues:
- `getDashboardMetrics` returns an all-zero `empty` object with `configured: false` when env missing or orders query fails.
- failed SMS count, inventory count, waste count are best-effort and coalesce to zero.
- `getOperationsIntelligence` can return demo-order fallback intelligence when env missing.
- `getShopIntelligence` composes multiple loaders that collapse errors into null/empty/zero.

Required V12.7 direction:
- centralise an `OperationalSnapshotV1` result for `/admin` and `/admin/today`.
- include `asOf`, branch id, and typed data state.
- distinguish `NO_DATA` from `UNAVAILABLE` and `CONFIGURATION_REQUIRED`.

## 4. Today Page Data Loaders

`/admin/today` uses:
- `getOwnerBrain`
- `getShopIntelligence`
- `getDashboardMetrics`
- `getOperationsIntelligence`
- `getInventoryBatches`
- `getPurchasingPlan`
- `getAllProducts`
- `getProductCostMap`

Current truth issue:
- Today and Admin calculate overlapping operational truth through separate chains.
- If a lower loader returns empty/zero on failure, Today can render "nothing urgent" rather than "data unavailable".

Required V12.7 direction:
- make Today consume the same operational snapshot as `/admin` where possible.
- preserve existing UI, but show an honest degraded/unavailable/configuration state at the top when snapshot state is not healthy.

## 5. Business Insights Data Loaders

Business Insights is `/admin` plus:
- `BusinessInsightsSections` from `getShopIntelligence`
- operations-intelligence detail panels

Current truth issue:
- analysis can be computed over mixed real/empty/fallback data.
- `dataState.status = "error"` exists in operations-intelligence but is not consistently elevated into an owner-visible page truth state.

Required V12.7 direction:
- use `OperationalSnapshotV1` state.
- render honest page state when data is degraded/unavailable.

## 6. Inventory / Catalog / Order Fallback Loaders

Inventory:
- `getSuppliers()` returns `[]` on env/query failure.
- `getInventoryBatches()` returns `[]` on env/query failure.
- V12.5 discovery notes this is not demo fallback, but it still conflates no data with unavailable.

Catalog:
- catalog functions return demo products/categories/settings on missing env or query failure.

Orders:
- counter orders return demo orders on missing env or query failure.
- public checkout mutation already fails closed when service env is missing.

Required V12.7 direction:
- replace production demo fallback with typed `DataResult`.
- where old array-returning functions are retained for compatibility, add result-returning variants and make production callers use those variants.

## 7. Empty-Array-as-Success States

Current empty-array-as-success examples:
- `getSuppliers()`
- `getInventoryBatches()`
- `getAuditEvents()`
- `getPurchasingPlan` internal source reads
- `getCarcassIntakes()`
- `getShopClosures()` on query failure
- `getOrderNotes()` on query failure

Not all are V12.7 launch blockers. For this slice, focus on owner-facing operational truth:
- admin dashboard
- Today
- Business Insights
- catalog/public branch/storefront
- counter orders
- releases
- inventory/catalog/order reads feeding operational snapshot

## 8. Production / Dev / Test Switches

Current switches:
- `NODE_ENV === "production"` is already used for staff/order-access secret fallbacks.
- `CHECKOUT_TEST_MODE_ENABLED` and `NEXT_PUBLIC_CHECKOUT_TEST_MODE` are explicit test-order controls.
- no explicit `DEMO_MODE` / `ALLOW_DEMO_DATA` switch exists for loader fallbacks.

Required V12.7 direction:
- production must not silently use demo data.
- dev/test may use demo data only through explicit non-production helper behavior.
- add a small helper like `isProductionRuntime()` / `allowDemoFallback()`.

## 9. Recommended V12.7 Implementation Shape

1. Add `DataResult`:
   - states: `HEALTHY`, `NO_DATA`, `DEGRADED`, `UNAVAILABLE`, `UNAUTHORISED`, `CONFIGURATION_REQUIRED`
   - helpers for healthy/noData/degraded/unavailable/configurationRequired

2. Add explicit storefront branch result:
   - `getPublicBranchResult()`
   - production requires `CANONICAL_BRANCH_ID` or `NEXT_PUBLIC_CANONICAL_BRANCH_ID`
   - no production first-active/demo fallback
   - keep `getPublicBranch()` only as a compatibility wrapper if needed, but update production callers.

3. Add result variants for high-risk loaders:
   - catalog products/categories/settings
   - pickup windows
   - counter orders
   - dashboard metrics
   - operations intelligence
   - releases governance
   - inventory batches/suppliers where used by snapshot

4. Add `OperationalSnapshotV1`:
   - branch id
   - `asOf`
   - data result state
   - dashboard metrics
   - operations intelligence
   - shop intelligence / owner brain as needed
   - issue list for degraded/unavailable sources

5. Update `/admin` and `/admin/today`:
   - consume snapshot state
   - show honest page-level messages for no data, unavailable, degraded, configuration required
   - avoid recalculating conflicting branch/time truth

6. Update public storefront pages:
   - if branch/config missing, render configuration/unavailable message rather than demo products.
   - no demo product/category fallback in production.

7. Add `scripts/verify-operational-truth.mjs`.

## 10. Risk Map

High:
- changing `getPublicBranch` affects storefront, checkout, product pages, SMS, header, and settings.
- propagating `DataResult` everywhere can become too broad.

Medium:
- existing tests may assume demo fallback when env is missing.
- admin UI needs concise honest states without becoming noisy.
- local dev convenience should remain explicit.

Low:
- typed wrappers add boilerplate but reduce ambiguity.

## 11. Stop / Go

GO for a narrow V12.7 implementation:
- typed `DataResult`
- production-safe canonical branch result
- no production demo fallback in catalog/pickup/order/release/operations paths
- central `OperationalSnapshotV1` for `/admin` and `/admin/today`
- honest UI state banners
- verification script

STOP: discovery complete. Implementation may begin only after this report exists.
