# V12.9 Discovery - Production Readiness

Date: 2026-06-07
Scope: V12.9 discovery only. No real production restores. No behavioural change.
Builds on: V12.0-V12.8.

## 0. Verdict

The application is hardened (V12.1-V12.7) and now observable (V12.8). What is
missing is the **operational scaffolding to run it safely in production**: a single
authoritative list of required secrets with an automated check, an explicit
migration-parity gate runnable as a readiness step, documented backup/restore
procedures, and a release-readiness checklist.

V12.9 adds these as controls and documentation. No new runtime feature; no real
production data is touched.

## 1. Environments

| Environment | Runtime | Database | Notes |
|---|---|---|---|
| Local dev | `next dev` (Turbopack), `NODE_ENV=development` | Local Supabase stack (`supabase start`, API on `:54321`) | Insecure dev secret fallbacks permitted; demo data allowed. |
| Preview | Vercel preview (`VERCEL_ENV=preview`) | Linked Supabase project | Should mirror production config; no demo fallback. |
| Production | Vercel (`VERCEL_ENV=production` / `NODE_ENV=production`) | Supabase cloud project | No insecure fallbacks; canonical branch required; demo data forbidden. |

`isProductionRuntime()` (`src/lib/server/runtime-truth.ts`) treats
`NODE_ENV=production` OR `VERCEL_ENV=production` as production. Deployment is via
Vercel (`.vercel/` present); environment files (`.env.local`,
`.env.production.remote`, `.env.vercel.*`) are gitignored and never bundled
(`scripts/audit-bundle.mjs` denylist + secret scan).

## 2. Secrets

Authoritative env surface is `.env.example`. Classification for production:

### Required (production fails closed without these)
- `NEXT_PUBLIC_SUPABASE_URL` — Supabase project URL.
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` — public anon key (RLS-bound).
- `SUPABASE_SERVICE_ROLE_KEY` — service-role key; the ONLY RLS-bypassing
  capability, server-side only. Without it checkout cannot write
  (`submitCheckout` returns 503) and security audit cannot emit.
- `ORDER_ACCESS_SECRET` — >= 32 bytes. Signs the public order-access cookie,
  salts rate-limit identity hashing, and is the staff-session fallback secret.
  Production throws without it (`order-access-session.ts`, `rate-limit.ts`).
- `STAFF_SESSION_SECRET` — >= 32 chars. Signs the staff session envelope. Falls
  back to `ORDER_ACCESS_SECRET` if unset, but a dedicated value is recommended.
  Production throws if neither is set/valid (`staff-session.ts`).
- `CANONICAL_BRANCH_ID` (or `NEXT_PUBLIC_CANONICAL_BRANCH_ID`) — storefront branch.
  V12.7 returns `CONFIGURATION_REQUIRED` in production without it.

### Conditionally required
- `TWILIO_ACCOUNT_SID` / `TWILIO_AUTH_TOKEN` / `TWILIO_FROM_NUMBER` and
  `SMS_SENDING_ENABLED=true` — only when real SMS is enabled. Default OFF; SMS is
  honestly recorded as "disabled" otherwise (`sms.ts`, `launch-readiness.ts`).

### Optional
- `ALERT_WEBHOOK_URL` — V12.8 alert webhook sink.

### Must NOT be enabled in production (hygiene)
- `ALLOW_DEMO_DATA` — must be unset/false; `true` re-enables demo fallback.
- `CHECKOUT_TEST_MODE_ENABLED` / `NEXT_PUBLIC_CHECKOUT_TEST_MODE` — must be off so
  no test orders can be created.
- `MIGRATION_DRIFT_CHECK_MODE` — must NOT be `local`/`dev` in CI/release.
- `RELEASE_REPORT_MODE` — `local` only for local runs.

Gap: there is no single automated check that all required secrets are present and
valid. V12.9 adds `scripts/verify-production-readiness.mjs`.

## 3. Deployments

- Hosting: Vercel. Build: `next build`. Middleware bundles the edge auth/audit path.
- Promotion: preview → production via Vercel. Env vars managed in Vercel project
  settings (mirrored locally only in gitignored `.env.vercel.*`).
- No custom deploy scripts in-repo beyond the Next build; deploy orchestration is
  Vercel-native.

## 4. Migrations

- Source of truth: `supabase/migrations/*.sql` (timestamp-prefixed, 20 files
  through `202606071700_v12_6_checklist_evidence_integrity.sql`).
- Apply: Supabase CLI (`supabase db push` / linked project), tracked in
  `supabase_migrations.schema_migrations`.
- Parity tooling: `scripts/check-migrations.mjs` (release drift check via
  `get_applied_migration_versions` RPC, CLI fallback) and the DB function
  `get_migration_health()` (required-migration applied status, anon-readable).
- Risk: schema drift between repo and the production DB. V12.9 makes parity an
  explicit readiness check; V12.8 `/api/health` surfaces it at runtime.
- Note: V12.5/V12.6 added migrations; the operator must `supabase db push` before
  those features work in production (per memory: prod DB push needed first).

## 5. Backups

- Supabase cloud provides automated daily backups; Point-in-Time Recovery (PITR)
  is available on paid tiers. This is the primary backup mechanism — the app does
  not implement its own backup pipeline.
- Gap: no documented backup verification or restore drill. V12.9 adds
  `docs/runbooks/backup.md` and `docs/runbooks/restore.md` with validation steps
  (no real production restore is performed).

## 6. Restore Process

- Restore is performed through the Supabase dashboard (Database → Backups) or via
  PITR to a timestamp, optionally into a fresh project for validation before
  cutover.
- After any restore, migration parity and the V12.8 health endpoint must be
  re-verified, and a smoke test (login, shop, checkout, counter, audit) run.
- Documented in `docs/runbooks/restore.md`. No production restore is executed in
  this phase — validation steps only.

## 7. Recommended V12.9 implementation shape

1. `scripts/verify-production-readiness.mjs`:
   - verify required secrets present + length (service role, ORDER_ACCESS_SECRET,
     STAFF_SESSION_SECRET-or-fallback, public url/anon, canonical branch);
   - production hygiene (demo/test flags off);
   - migration parity vs DB when reachable;
   - strict mode (production/CI: missing => FAIL); local mode (dev: missing
     production-only secret => SKIP, so the cert gate is runnable locally),
     mirroring `check-migrations.mjs` and `verify-checkout-integrity.mjs`.
2. `docs/runbooks/backup.md`, `docs/runbooks/restore.md` — procedures + validation.
3. `docs/releases/release-readiness-checklist.md` — migrations, health, auth,
   checkout, inventory, audit, backups.
4. Add `CANONICAL_BRANCH_ID` to `.env.example` (currently undocumented there).
5. `docs/releases/v12.9-validation-report.md`.

## 8. Risk map

High: none — controls + docs only.
Medium: production-only secret validation cannot fully pass in a dev environment
without the real secrets; the script SKIPS those locally and is FAIL-strict under
`PRODUCTION_READINESS_MODE=strict` / `NODE_ENV=production`. The certification
report must record that strict secret validation is a production-only step.
Low: backup/restore are operator-executed Supabase procedures; this repo documents
and validates them but cannot exercise a real production restore.

## 9. Stop / Go

GO for a narrow V12.9 implementation: readiness verifier, backup/restore runbooks,
release-readiness checklist, `.env.example` completion, validation report.

STOP: discovery complete. Implementation may begin now.
