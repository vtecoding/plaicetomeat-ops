# V12.4 Discovery — Audit Semantics Inventory

Date: 2026-06-07
Scope: **discovery / inventory only.** No implementation, no migrations, no test edits.
Builds on: V12.1 (RPC seal), V12.2 (identity/branch), V12.3 (checkout integrity).
Purpose: answer the six audit questions so V12.4's size can be judged before any code.

---

## 0. Verdict (read this first)

**Audit authenticity and transactionality are strong; audit *coverage* and *path
consistency* are the gaps.** Specifically:

- ✅ Direct client writes to `audit_logs`/`audit_events` are sealed (V11.2).
- ✅ Indirect emission by anon/authenticated is impossible — `emit_audit_log` is
  `service_role`-only (V12.1).
- ✅ **100% of audit rows are produced inside a business-RPC transaction** — there
  are *zero* app-layer emit calls — so every audit row is atomic with its mutation.
- ⚠️ **`security_event` has no producer at all.** The whole V12.2 identity/session/
  branch-authority surface (failed logins, lockouts, tampered/cross-user sessions,
  unauthorised access, logout failures) writes **no audit evidence**.
- ⚠️ **Two emission doors with unequal guarantees.** The hardened `emit_audit_log`
  (allowlist + secret-stripping + size cap) is **dead code**; the authoritative path
  is ~30 inline `INSERT INTO audit_logs` statements that bypass those hygiene checks.
- ⚠️ A few business writes emit nothing (e.g. `add_order_note`).

**So V12.4 is SMALLER than a rewrite.** It is mainly: (1) wire `security_event`
emission for the V12.2 auth events, and (2) reconcile the two emission doors
(make inline inserts conform, or route them through one helper). Authenticity does
not need re-sealing.

## 1. Audit data model

```
business RPC (SECURITY DEFINER, in txn)
   ├─ INSERT INTO audit_logs (...)        ← authoritative door (≈30 sites)
   └─ PERFORM emit_audit_log(...)         ← only transition_order_status uses it
                          │
emit_audit_log (service_role only)  ──────┘  hardened door: allowlist + secret-strip + size cap
                          │
                          ▼
                    audit_logs  (canonical, append-only)
                          │  AFTER INSERT trigger: mirror_audit_log_to_event (SECURITY DEFINER, same txn)
                          ▼
                    audit_events (denormalised read-model: actor_email/role + summary)
                          │
                          ▼
              /admin/audit  (getRecentAuditEvents → audit_events)
```

- `audit_logs`: canonical evidence (`event_type, target_type, target_id, branch_id, actor_id, metadata, created_at`), append-only (`audit_logs_append_only`).
- `audit_events`: denormalised mirror the UI reads, populated **only** by the trigger (direct insert revoked V11.2), append-only.
- App transport: [`emitAuditLog`/`emitSecurityEvent`](src/lib/server/audit.ts) (service-role) — **imported only by its own test; never called in production code**.

## 3. Q2 / Q3 — Authoritative vs generic

Definition used: **authoritative** = emitted inside a trusted business RPC, in the
same transaction as the mutation, actor bound server-side. **generic** = emitted via
the app-layer `emit_audit_log` service-role transport (out-of-band, best-effort).

Because there are **no app-layer emit calls**, **30/31 event types are authoritative**
and the only "generic" type (`security_event`) is **currently unproduced**.

## 4. Q4 — Which can still be emitted indirectly?

**None by untrusted callers.** Verified (and re-confirmed by `verify-audit-authenticity.mjs`):
- `audit_logs`/`audit_events` have **no anon/authenticated INSERT/UPDATE/DELETE** and no insert/all RLS policy (V11.2).
- `emit_audit_log` EXECUTE is **revoked from anon + authenticated**, granted to `service_role` only (V12.1).
- `audit_events` is writable **only** by the `mirror_audit_log_to_event` trigger (owner context).
- Append-only triggers block UPDATE/DELETE on both tables.

The only "indirect" producer is the **mirror trigger**, which is system-owned and
fires only on a legitimate `audit_logs` insert. No forgery surface remains.

## 5. Q5 — Which business operations produce NO audit evidence?

| Operation | Audit? | Notes / severity |
|---|---|---|
| **Auth: failed login** | ❌ | `loginAction` logs to console only; no `security_event`. **High** |
| **Auth: account/network lockout tripped** | ❌ | no evidence a lockout occurred. **High** |
| **Session: tampered / cross-user / expired envelope rejected** (V12.2 middleware) | ❌ | silent `signOut`+redirect; no `security_event`. **High** |
| **Authority: `/unauthorised` (role/branch refusal)** | ❌ | `requireStaffContext` refusals unlogged. **Medium** |
| **Auth: logout failure** | ❌ | surfaced to user (V12.3) but not audited. **Low** |
| **Action-layer branch denial** (`requireBranchStaff`, etc.) | ❌ | refusals unlogged. **Medium** |
| `add_order_note` | ❌ | staff annotation; no `note_added` type exists. **Low** |
| checkout rate-limit / capacity rejections (V12.3) | ❌ | failures not audited (only successes). **Low/Medium** |
| All successful business mutations (orders, products, pricing, inventory, ops, pickup/closures, suppliers, releases, SMS) | ✅ | authoritative, in-txn |

**Theme:** every *successful state change* is audited; almost no *security/authority
failure* is. `security_event` exists precisely for this and is unused.

## 6. Q6 — Which audit events are emitted outside transactions?

**None.** Every audit row is written either by an inline `INSERT INTO audit_logs`
inside a business RPC, or by `PERFORM emit_audit_log(...)` inside
`transition_order_status` — both within the RPC's single transaction. The mirror
trigger fires `AFTER INSERT` **within that same transaction**. There are no
app-layer (`src/lib/server/audit.ts`) emissions, so there is no best-effort,
post-commit audit write anywhere.

Minor caveat (not an out-of-txn audit write): `record_sms_attempt` runs in its own
transaction after `transition_order_status`, wrapped in a swallowing try/catch in
[counter.ts]. If it fails, the `sms_attempt` audit is simply absent — the order
transition and its `order_status_changed` audit are unaffected. This is best-effort
*SMS bookkeeping*, by design, not a broken mutation audit.

## 7. Findings

| ID | Severity | Finding |
|----|----------|---------|
| A1 | **High** | `security_event` is defined but never produced; the entire V12.2 auth/identity/session/branch-authority surface emits no audit evidence. The sanctioned `emitAuditLog`/`emitSecurityEvent` helpers are dead code. |
| A2 | Medium | Two emission doors: the hardened `emit_audit_log` (allowlist + secret-strip + size cap) is unused, while ~30 inline `INSERT INTO audit_logs` bypass those checks. Authoritative but unenforced hygiene; risk of a future RPC logging a secret or an off-allowlist `event_type`. |
| A3 | Medium | No audit for authority refusals (`/unauthorised`, branch denials) or abuse rejections (rate-limit, capacity-full) — only successes are recorded, weakening incident investigation. |
| A4 | Low | `add_order_note` (and possibly other annotations) produce no audit row; no `note_added` vocabulary. |
| A5 | Low | `audit_events.summary` is a mechanical `replace(event_type,'_',' ')` — low-fidelity for the owner-facing log (semantics/readability, not integrity). |
| A6 | Info | `audit_events` is a denormalised snapshot (actor_email/role copied at insert) — by design; renames/role changes won't retro-update history. |

No P0. No authenticity hole. No forgery/indirect-emission surface.

## 8. Recommended V12.4 shape (for approval — not started)

1. **Emit `security_event` for V12.2 auth/authority events** (A1, A3): wire
   `emitSecurityEvent` (already built, service-role, system-reason) into
   `loginAction` (failed login, lockout), middleware (session invalid/expired —
   carefully, avoid noise), and `requireStaffContext`/`requireBranchStaff` refusals.
   Metadata: hashed identity only (reuse `hashIdentity`), never PII. This activates
   the dead-but-correct helper rather than building new infrastructure.
2. **Reconcile the two doors** (A2): either route business-RPC audit writes through
   a single internal SQL helper that enforces the allowlist + secret-strip, or add a
   migration-level assertion/trigger that rejects off-allowlist `event_type` and
   secret-like metadata keys on `audit_logs` directly. Forward-only.
3. **Optional**: `note_added` audit for `add_order_note` (A4); richer `summary` (A5).

**Migrations needed:** small/optional — only if reconciling the door at the DB
(A2) or adding `note_added` to the allowlist. The security-event wiring (A1/A3) is
**app-layer only**, no migration.

**Test targets:** extend `verify-audit-authenticity.mjs` (or a new
`verify-audit-semantics.mjs`) to assert: a failed login/refusal produces a
`security_event`; off-allowlist/secret metadata is rejected on the authoritative
door; every successful business RPC still emits exactly one authoritative row.

## 9. Readiness assessment

- The audit system's **integrity is sound**: authentic, append-only, in-transaction,
  un-forgeable, no indirect emission. The discovery-before-build instinct holds again
  — there is **no audit-authenticity emergency**.
- The real work is **coverage of security/authority events** (activating the dead
  `security_event` path) and **door consistency** — both contained, mostly app-layer,
  with at most a small forward migration.
- **Recommendation: V12.4 is a small/medium slice, not a critical phase.** Approve a
  focused V12.4 to (1) emit `security_event` for V12.2's auth surface and (2) reconcile
  the two emission doors; defer `note_added`/summary polish unless wanted.

**STOP.** Inventory only — no code changed. Awaiting approval before any V12.4 implementation.
