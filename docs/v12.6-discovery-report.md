# V12.6 Discovery - Checklist / Compliance Evidence Integrity

Date: 2026-06-07
Scope: V12.6 discovery only. No implementation, no migrations, no test edits.
Builds on: V12.1-V12.5.

## 0. Verdict

The guided checklist layer is good at provenance, idempotency, branch authority, and resumability, but it is not yet evidence-hard. The database stores sessions and append-only step events, but it does not bind a session to a versioned checklist definition, does not validate step keys against server-known definitions, does not validate evidence payload shape/ranges, and lets a checklist complete even when no required evidence exists.

Primary V12.6 integrity gaps:
- `ops_record_step` accepts any non-empty `step_key`.
- `ops_complete_session` completes any in-progress session without checking persisted required steps.
- `ops_checklist_sessions` has `kind` but no checklist definition id/version.
- `ops_checklist_events.payload` is arbitrary JSON.
- Receipts are rebuilt from current TypeScript definitions, so a future checklist edit can change how old evidence is displayed.
- Temperature evidence exists in two worlds: guided opening/closing number steps and the older demo-only counter compliance page.

Recommendation: GO for a narrow evidence-hardening migration that adds database-known versioned checklist definitions, binds sessions to a definition version, validates step keys/payloads in SQL, and blocks completion until required persisted steps are complete. Keep old receipts readable by making receipt rendering tolerate legacy sessions/events.

## 1. Checklist Tables

### `ops_checklist_sessions`

Created in `supabase/migrations/202606041700_v10_phase2_guided_capture.sql`.

Current columns:
- `id`
- `branch_id`
- `kind` with check `opening | closing | stock_count`
- `business_date`
- `status` with check `in_progress | completed | abandoned`
- `started_by`, `started_at`
- `completed_by`, `completed_at`
- `source`

Current integrity:
- RLS read policy for branch managers.
- Writes happen through SECURITY DEFINER RPCs.
- Unique in-progress session per `(branch_id, kind, business_date)`.

Missing:
- no `definition_key`
- no `definition_version`
- no immutable definition snapshot
- no completion metadata beyond `completed_at/completed_by`

### `ops_checklist_events`

Current columns:
- `id`
- `session_id`
- `branch_id`
- `step_key`
- `state` with check `done | skipped | na`
- `payload jsonb`
- `actor_id`
- `source`
- `idempotency_key`
- `created_at`

Current integrity:
- append-only in practice through RPCs; current table has no direct client write policy.
- retry-safe unique index on `(session_id, step_key, idempotency_key)` when key present.
- audited by `ops_step_recorded`.

Missing:
- no FK to a checklist definition step.
- no server-side step membership validation.
- no payload type/range validation.
- no required-step distinction.
- no state restriction per step.

### `stock_count_lines`

Stock count uses `ops_checklist_sessions(kind='stock_count')` as the session container, but evidence lives in `stock_count_lines`, not `ops_checklist_events`.

V12.5 now protects count apply from stale writes. V12.6 should not merge stock-count evidence into opening/closing checklist semantics unless discovery in implementation proves a small safe bridge is necessary.

## 4. Checklist Steps and Evidence

`GuidedChecklist` records:
- confirm steps: `payload = {}`
- number steps: `payload = { value: Number(numberValue) }`

`scripts/verify-ops-capture.mjs` currently records `fridge_temp` with `payload = { temp_c: 3.5 }`, which does not match UI shape. This proves the RPC currently accepts arbitrary payload shapes.

`buildReceipt` only displays payload details when:
- current TypeScript definition has the step key
- current definition input kind is `number`
- payload has numeric `value`

Therefore:
- old rows with `{ temp_c: 3.5 }` remain persisted but are not rendered as temperature detail by the current receipt builder.
- unknown step keys persist but disappear from receipts because receipts are definition-driven.

Current coupling:

- no conflict detection.
- no duplicate-write prevention.

V12.6 narrow recommendation:
- validate checklist temperature payloads strictly.
- do not claim temperature is merged into the older compliance model unless a small canonical write path is implemented and tested.
- document full temperature single-source merge as deferred if it exceeds the narrow integrity fix.

## 6. Completion Rules

Current SQL:
- `ops_complete_session` locks the session, checks auth/branch/status, then marks completed.
- It does not inspect `ops_checklist_events`.

Current UI:
- only shows Finish when all TypeScript-defined steps have client state.
- the server does not enforce this; direct RPC callers can complete an empty session.

Current risks:
- empty checklist can complete.
- missing critical/required steps can complete.
- unknown step events can make the event log look busy while required known steps are absent.
- completion audit is emitted even for an invalid/empty completion.

- `src/components/ops-capture/guided-checklist.tsx` for immediate client receipt after completion

Receipt currently includes:
- kind
- title
- completed-at label
- handled count
- total count
- step title/state/detail

Missing from receipt:
- definition key/version
- session id
- actor
- branch
- exact completion timestamp

Historical-read risk:
- receipts are rebuilt against current TypeScript definitions, not the definition that was active when evidence was recorded.

## 9. Arbitrary Step-Key Acceptance

Arbitrary keys are accepted in:
- `ops_record_step`: validates non-empty `p_step_key`, but no membership check.
- `scripts/verify-ops-capture.mjs`: currently proves invalid state rejection with step key `x`, but does not prove unknown key rejection because unknown keys are accepted if state is valid.

Impact:
- fake evidence can be appended under plausible or unknown keys.
- required evidence can be absent while unrelated steps exist.
- receipts omit unknown keys, so database evidence and visible receipt can diverge.

## 10. Empty / Incomplete Completion

Empty completion is accepted by:
- `ops_complete_session`: no event count or required-step check.

Incomplete completion is accepted by:
- same path; no per-kind required key list.

Completion audit currently emits after invalid completion because the RPC has no validity check.

## 13. Recommended V12.6 Implementation Shape

1. Add database-owned checklist definitions:
   - `ops_checklist_definitions`
   - `ops_checklist_definition_steps`
   - seed opening/closing version 1.

2. Bind sessions to definitions:
   - add nullable/filled `definition_id`, `definition_key`, or `definition_version` to `ops_checklist_sessions`.
   - backfill existing opening/closing sessions to version 1 where possible.
   - keep stock_count as a special session kind or define a separate stock_count v1 only if useful.

3. Harden `ops_start_or_resume_session`:
   - resolve active definition for `opening`/`closing`.
   - store definition/version on session.

4. Harden `ops_record_step`:
   - reject unknown step key for opening/closing.
   - validate state against allowed values.
   - validate payload shape by step input type.
   - for number steps require `{ value: number }` when state is `done`.
   - for confirm steps require empty object or no evidence payload.
   - enforce bounds:
     - Celsius: conservative food-safety range, e.g. `-30` to `30`.
     - GBP/cash: `0` to a high sane limit.
   - keep skipped/na as real states; consider requiring reason later, but not needed for V12.6 integrity.

5. Harden `ops_complete_session`:
   - lock session.
   - derive latest event per required step from persisted events.
   - reject empty/incomplete sessions.
   - emit `ops_session_completed` only after validation passes.

6. Receipt read compatibility:
   - include definition/version/session/branch/completion metadata in new receipts.
   - make read path tolerate legacy sessions with no definition metadata.
   - keep historical receipts readable even when old payload detail is absent.

7. Verification:
   - add `scripts/verify-checklist-integrity.mjs` or extend `scripts/verify-ops-capture.mjs`.
   - update existing `verify-ops-capture` payload for `fridge_temp` to `{ value: 3.5 }` if SQL begins validating payloads.

## 14. Risk Map

High:
- Definition drift between TypeScript and SQL. Mitigation: keep version 1 small and explicit; add tests that TS keys and SQL-seeded keys match.
- Breaking historical receipts. Mitigation: legacy-tolerant reads; enforce only write-time validation.
- Completion rule accidentally blocks stock_count sessions. Mitigation: scope required-step checks to opening/closing or define stock_count behavior separately.

Medium:
- Temperature bounds too strict. Mitigation: choose conservative operational bounds and document them.
- Current `verify-ops-capture` uses `{ temp_c }`; update the verifier when strict payload validation lands.
- UI immediate receipt may diverge from server receipt. Mitigation: include the same definition metadata and use persisted receipt on refresh.

Low:
- Extra joins on checklist reads. Not a correctness risk at shop scale.

## 15. Stop / Go

GO for a narrow V12.6 implementation:
- versioned DB definitions for opening/closing v1
- session binding to definition/version
- unknown step rejection
- required-step completion enforcement
- typed/bounded payload validation
- receipt metadata expansion with legacy readability
- verification coverage

Do not implement the full temperature single-source compliance merge unless it is clearly low-risk during implementation. If it is not low-risk, document it as deferred with explicit rationale.
